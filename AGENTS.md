## 工作流程（触发条件 + skill 映射）

执行细节由各 skill 负责，本表只列"什么时候用什么"。skill 安装在 `.ai/skills/`，详见 `.ai/TOOL-MAPPING.md`。

| 阶段 | 触发条件 | 用的 skill / agent |
|------|---------|-------------------|
| 新版本启动 | 用户声明"开始 V1.0.X 迭代"（AI 不主动建议） | `project-iteration` |
| 需求撰写 | 写 PRD | `.ai/agents/prd-writer`（专用 agent） |
| 方案设计 | 写技术方案 / 业务方案 | `.ai/agents/solution-designer`（专用 agent） |
| 规格文档 | 改原型前 / 新建页面规格 | `proto-spec-generator` |
| 原型设计 | 画原型 / 改原型 | `prototype-html-pin` |
| 评审 | 评审会结论落地 | 记入 `06-变更记录/`，被评审文档标注修改来源 |
| TAPD 上传 | 需求要进 TAPD | `tapd-requirement-upload` |
| 文档站发布 | 原型改完（必须同步，不允许脱节） | `vitepress-deploy` |
| 材料入库 | 新文件/会议记录进项目 | `project-intake-pm` |
| 版本收尾 | 版本结项 | `project-iteration`（校正 _index + 更新总览） |
| 复盘 | 收尾后主动提醒 | `project-retrospective-pm` |

需求撰写纪律：默认只写 1 份 PRD（含背景/范围/字段/交互）；复杂需求按需追加方案设计；增量需求按主题命名，不用子版本号叠加。

改原型时两种模式：用户没说新版本则原页面直接改；声明了新版本则剥离干净骨架后只对新改动加标注。改完用 `_scaffold/` 的页面跑 build.sh，交付前跑 check-annotations.sh。

## 文档生成纪律（撰写/编辑/输出/查询必守）

### 生成时

1. **完整性校验**：从源文档抽取列表或表格时，逐条对照条目数量。源文档有 N 项，产出不得少于 N 项；如有合并或省略，必须显式说明原因，不允许无声丢项。
2. **示例不等于定义**：源文档中标记为"如""举例""假设"的数值属于示例，汇报中必须还原为规则描述（如"按配置比例计算"而非"按20%计算"），不得将示例数值写死为定义。
3. **最小信息密度**：每个功能点至少保留"做什么 + 怎么用/为什么"两层信息，不允许只写功能名词不加解释。判断标准：不了解项目背景的读者能否看懂该功能在解决什么问题。
4. **角色必带边界**：每个角色首次出现时，用一句话交代"能做什么 + 不能做什么"，尤其当多角色共享系统时，边界不清会导致理解混乱。
5. **流程闭环校验**：写操作流程时，检查每个功能是否有"触发条件、操作步骤、执行结果"的完整闭环。如果发现"用户导入了数据，但没有后续操作让数据产生价值"，说明漏了桥接功能。

### 编辑时

6. **编号一致性**：插入、删除、移动编号项（如 F1.1、3.2.1）后，必须全局刷新后续编号和所有交叉引用，不允许出现编号跳跃或引用指向旧编号。
7. **编辑后自查**：每次编辑完成后，自查是否有重复/残留段落、是否有断链引用、是否有编号不连续。发现即修，不留给用户发现。
8. **中文编码处理**：解压含中文文件名的 zip 时，优先使用 cp437 转 gbk 解码方案，避免乱码。
8.1. **原型↔规格↔需求联动**：改了原型（含交互细节、状态、字段、文案），必须主动同步对应的规格文档和需求文档（涉事功能点），同步完成后明确告知用户改了哪几份文档。**不允许只改原型就结束、等用户下一轮提醒才补规格/需求。** 判断范围：原型里动了什么模块，就回规格/需求里找同一功能点对齐。
8.2. **规格模块覆盖校验**：写或改页面规格时，对照该页原型里实际存在的所有模块逐个核对，不允许只写主模块漏掉附属模块。发现历史规格漏模块时，补全而非视而不见。
8.3. **文件定位不靠推断**：找文档路径时用 ls/find 确认实际位置，不要凭原型 HTML 的目录路径推断规格/需求文档在同目录（规格常在父级或同级 03-原型设计/ 下）。

### 输出时

9. **Word 默认排版**：生成 docx 时使用以下规范，全局微软雅黑、正文 10.5pt、标题色 #1A3C6E、表格表头深蓝底白字（#1A3C6E / #FFFFFF）、数据行白蓝交替（#FFFFFF / #EEF2F8）、行距 1.35。

### 命名时

11. **页面编号前缀统一**：生成页面清单、规格文档、原型时，页面编号前缀只表达运行端、不表达角色，统一 2 位大写字母（PT 门户 / WS 工作台 / AD 管理后台 / MP 小程序 / AP APP端 / H5 通用移动端 / OT 其他端）。完整规则见 `01-通用规则/项目工作流规范.md` 第十五节。老项目既有编号不回改。

生成完成后，按上述 11 条逐条自查。发现遗漏立即修正，不需要等用户指出。

### 查询时

12. **先查知识库**：遇到 B 端 SaaS 的角色设计、权限设计、注册流程问题时，先查知识库 `B端产品/` 目录下的模式笔记再回答，不要凭直觉设计。

## 硬约束（AI 必须遵守）

1. **进入项目先读 `00-项目总览.md` 和 `02-迭代/_index.md`**
2. **新建/改需求前**：先读 `01-项目基线/关键决策.md` 确认前置约束
3. **画原型前**：先读对应规格文档，确认功能范围
4. **收到二进制文件**：先转 MD 存 `_source/`，优先读 `_转换.md`
5. **引用关键决策**：用 `#D001` 编号格式
6. **交付前**：自查是否标注决策来源和需求依据
7. **跨版本交接**：上一版资产保留在原版本目录，新版本复制使用
8. **协作方式**：务实、批判性、结构化，跳过客套直接动手。该质疑就质疑，该推荐就推荐
9. **原型改动必须三方联动**：改原型后主动同步规格+需求（见 8.1），不等用户提醒
10. **调研类任务前置确认**：做竞品调研/市场分析前，必须先确认调研主题、客户场景假设、报告用途、决策定位四项，未确认不动手（详见 `competitor-analysis-pm` skill 阶段零）。客户场景假设错误是整篇返工的最大来源
11. **绝对化结论前搜反例**：准备写"X 没有 Y""市面零 Z"这类绝对化论断时，必须先主动搜一次反例，搜到则收窄论断
12. **差异化卖点要判定**：声称"这是我们的差异化"前，先问"这是我方独有还是对方也有"，区分平台差异化与厂商自带能力
13. **产品项目名用全称**：全文首次出现用全称，用户纠正过名称后立即全文统一
14. **产出物类功能前置确认**：设计"导出/报告/下载/生成文件"类功能时，动手前必须先确认三项——给谁看（汇报对象）、做什么用（存档/对比/演示）、产出物形态（必须生成文件，还是在线预览够）。默认推荐轻量方案（浏览器打印/截图/在线预览），重方案（服务端生成文件）作备选并列对比成本，不作为默认推荐。方案敲定前不动规格和原型，不在方案未定时改文档
15. **对外材料只放客户可感知能力**：生成销售话术、对外宣讲、报价单等面向外部客户的材料时，判断标准是"客户登录 SaaS 后能直接看到、用到、感受到的能力"，不是按"前端/后端"粗分。计费套餐、积分池等产品形态虽是客户购买的，但属于平台运营层设计，非客户侧直接操作的功能，不列为宣讲卖点；代理商分销、财务账务、运营后台等内部基建同理不列。合作方/供应商具体名称用概括性表述替代，除非用户明确授权公开

## 写作风格约束

- 客户级正式文档禁止使用箭头（→）、等号（=>）等非正式符号，用逗号、分号或破折号衔接
- 功能描述采用自然段落叙述，禁止"流程一/流程二"机械编号
- 全中文专业沟通，输出对齐标准原文规范
- 功能描述 ≥67 字，控制在 120 字以内，阿拉伯数字分点
- 禁止使用 emoji 图标
