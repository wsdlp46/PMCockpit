# 项目工作流规范

_最后更新：2026-07-02 | 版本：V2.1（合并知识库指南，统一为单一权威文档）_

本规范定义了本工作区中每个项目从 0 到 1 的标准操作流程、文件关联方式、知识库治理和 AI 协作行为准则。原《项目知识库搭建指南.md》已并入本文件，不再单独维护。

> **V2.1 变更要点**：合并原《项目知识库搭建指南.md》，消除两份文件的重叠和矛盾；删除已废弃的 .workbuddy/memory 记忆系统和 WorkBuddy 提示词引用；执行细节统一引用对应 skill，不在本规范重复抄写步骤。

## 一、四层目录结构

```
项目名/
├── 00-项目总览.md              # 项目仪表盘（状态、迭代索引、关键决策索引）
├── _index.md                   # 项目文件全量索引（AI 进入项目的第一眼）
├── 项目工作流约定.md            # 本项目专属补充约定（可选）
│
├── 01-项目基线/                # 【第一层】跨版本累积性资产
│   ├── _index.md
│   ├── 关键决策.md             # 跨阶段提炼的设计约束（#D001 自增编号）
│   ├── 需求池.md               # 所有候选需求的统一入口
│   ├── 业务规则/               # 结算规则、分润规则等永久约束
│   ├── 设计规范/               # design.md + 原型样式案例 + 品牌色/Logo
│   ├── 架构图/                 # 系统架构、脑图、页面流转图
│   ├── 法律协议/               # 合规约束
│   ├── 会议记录/               # 按时间组织的外部输入（时间线 + 原始纪要）
│   ├── 项目工具/               # 本项目的流程规范、模板
│   └── _source/                # 基线相关二进制原件
│
├── 02-迭代/                    # 【第二层】所有版本迭代的容器
│   ├── _index.md               # 迭代总索引（版本清单 + 状态）
│   └── 0X-V1.0.X/              # 【第三层】单个版本，完整自包含
│       ├── _index.md           # 版本索引（导航 + 状态 + 需求清单）
│       ├── 01-原始需求/        # 本版本的外部输入材料
│       ├── 02-需求文档/        # PRD + 增量需求文档 + 功能清单
│       ├── 03-规格文档/        # 原型设计规格（按页面编号）
│       ├── 04-原型/            # HTML 原型 + _scaffold/ 脚手架
│       ├── 05-TAPD上传规划/    # TAPD 需求上传规划
│       └── 06-变更记录/        # 本版本的需求变更清单、差异清单、评审纪要
│
├── 03-对外材料/                # 汇报 PPT、产品介绍、白皮书
├── _source/                    # 临时性二进制文件归档
└── 归档/                       # 已废弃的页面、规格、方案
```

> 注：原 V1.0 规范的 `01-项目输入` 已拆分——外部输入的累积性部分（关键决策、会议记录）并入 `01-项目基线/`，单版本的原始输入进 `版本/01-原始需求/`。原 `04-架构图`、`05-评审纪要`、`06-对外材料` 重组：架构图入基线，评审纪要并入变更记录，对外材料提升为顶层目录。

## 二、双层知识库体系

本工作区包含两层知识库，分工不同：

| 层级 | 位置 | 内容 | 共享范围 | 生命周期 |
|------|------|------|---------|---------|
| **工作区级（大知识库）** | `01-通用规则/` + `03知识库/` | 通用模板、设计规范、方法论、案例库、政策法规、竞品分析 | 所有项目共享 | 持续累积，不随项目结束归档 |
| **项目级（项目知识库）** | `02当前项目/<项目名>/` | 本项目的基线、需求、原型、评审、对外材料 | 本项目内 | 随项目启动创建，结束后沉淀案例到工作区级 |

### 两层的交互

**启动阶段：从工作区级拉取**

| 场景 | 拉取内容 | 存放位置 |
|------|---------|---------|
| 项目启动 | 复制 PRD 模板、方案模板 | 项目 `02-需求文档/` |
| 需求撰写 | 引用术语表定义、政策法规标准 | 项目文档内链接引用 |
| 原型设计 | 引用设计规范、设计库组件 | 原型规格中标注规范依据 |
| 评审准备 | 使用检查清单 | `06-变更记录/` 中引用 |
| 竞品分析 | 查阅 `03知识库/竞品分析/` 已有报告 | 需求文档中引用结论 |

**结束阶段：反向沉淀到工作区级**

| 沉淀内容 | 目标位置 | 触发条件 |
|---------|---------|---------|
| 可复用设计模式 | `01-通用规则/设计库/` | 项目中验证有效且复用 2 次以上 |
| 踩坑经验 | `01-通用规则/方法论/` | 复盘后判断有通用价值 |
| 完整项目案例 | `03知识库/工作案例库/` | 项目复盘后执行 |
| 新术语 | `01-通用规则/术语表.md` | 项目中首次定义且跨项目通用 |
| 新检查项 | `01-通用规则/检查清单/` | 评审中发现既往清单未覆盖 |

## 三、通用 vs 迭代的分界线（关键纪律）

决定一个文件该放 `01-项目基线/` 还是 `02-迭代/版本号/`：

| 01-项目基线/ | 02-迭代/版本号/ |
|---------|---------------|
| 跨版本累积性资产（关键决策、需求池、竞品分析、路线图） | 本版本专属产物 |
| 永久约束（品牌、法律协议、业务规则） | 需求文档、方案设计 |
| 系统级资产（架构图、脑图、页面流转图） | 规格文档、原型文件 |
| 按时间组织的资产（会议记录） | TAPD 上传规划、变更记录 |
| 项目工具和流程规范 | 原始需求材料 |

**核心原则**：基线中的内容应是版本的可靠输入，不应包含未确定的实验性产物。迭代中产生的脑图、流程图、业务规则，先存当前版本的 `02-需求文档/` 或 `06-变更记录/`，确定不再变动后再合并到 `01-项目基线/`。

## 四、版本迭代规则

### 4.1 版本号与命名

- 版本文件夹：`0X-V{MAJOR}.{MINOR}.{PATCH}`，如 `01-V1.0.0`、`02-V1.0.1`
- 关键文件（需求文档、TAPD 规划）文件名带版本号：`xxxV1.0.0.md`
- 原型文件和规格文件不带版本号，版本信息在路径上

### 4.2 版本资产（每个版本完整自包含）

每版本文件夹包含完整可交付成果：`01-原始需求` 到 `06-变更记录` 六个子目录。评审纪要不单设目录，结论作为变更条目记入 `06-变更记录/`。

### 4.3 初始化与收尾

版本的初始化（建目录骨架、复制上版资产、剥离原型标注）和收尾（校正 _index、更新总览）由 `project-iteration` skill 执行。**执行细节见该 skill，本规范不重复。** 触发方式：对 AI 说"开始新版本"或"版本收尾"。

## 五、工作流阶段

```
版本规划 → 需求撰写 → 规格+原型 → 评审 → TAPD上传 → 文档站发布 → 版本收尾+复盘
```

### 阶段 1：版本规划
1. 同步需求池到 `01-项目基线/需求池.md`
2. AI 读取需求池 + 关键决策，输出候选范围
3. 人确认后对 AI 说"开始新版本"（触发 `project-iteration` skill 建目录）
4. AI 生成 `06-变更记录/需求范围清单_VX.X.X.md`

### 阶段 2：需求撰写
1. 参考 `01-项目基线/关键决策.md`、`业务规则/`、`架构图/`
2. 写入 `02-需求文档/`

**需求文档层次（精简原则）**：
- 默认只写 1 份 PRD（含背景/范围/字段/交互）
- 复杂需求按需追加方案设计文档，默认不写，避免与 PRD 重复
- 每批次新需求出独立的增量需求文档（按主题命名），不用子版本号叠加
- 基线需求文档（如 MVP V1.0）内容冻结，变更以增量方式叠加

需求撰写的执行细节（查案例库、匹配模式、跑检查清单）由 `prd-writer` agent 执行。

### 阶段 3：规格 + 原型
1. 参考 `01-项目基线/设计规范/` 取主色和样式
2. 参考 `01-项目基线/法律协议/` 检查合规
3. 规格写入 `03-规格文档/`，原型写入 `04-原型/`

**规格文档命名规范**：
- 新建页：`XX-页面名-原型设计规格.md`（完整规格）
- 变更页：`XX-页面名-变更规格.md`（仅变更描述）
- 规格文档头部维护变更记录头（倒序），正文原地修改

规格生成由 `proto-spec-generator` skill 执行，原型生成由 `prototype-html-pin` skill 执行。

### 阶段 4：评审
- 评审结论作为变更条目记入 `06-变更记录/`
- 被评审文档标注修改来源：`修改依据：[[06-变更记录/xxx#评审项N]]`

### 阶段 5：TAPD 上传
- 生成 `05-TAPD上传规划/TAPD上传规划_VX.X.X_{端}.md`
- 按规划逐条上传，由 `tapd-requirement-upload` skill 执行

### 阶段 6：文档站发布
- 使用 `vitepress-deploy` skill 将原型同步到文档站
- 原型改完后必须同步，不允许源项目与文档站脱节

### 阶段 7：版本收尾 + 复盘
1. 对 AI 说"版本收尾"（触发 `project-iteration` skill 校正 _index、更新总览）
2. 上一版本带标注原型保留在原版本目录，不额外归档
3. 输出汇报材料到 `03-对外材料/`
4. 执行复盘（对 AI 说"复盘"，触发 `project-retrospective-pm` skill），覆盖提示词缺陷、知识库缺失、可技能化的重复流程

## 六、材料入库与决策提炼

> **执行细节由 `project-intake-pm` skill 负责，本节只讲什么时候触发、产出放哪里。**

### 6.1 触发条件

收到以下任一情况时，对 AI 说"入库"或"整理一下"：
- 新文件（PDF/DOCX/XLSX/PPTX/图片）
- 会议内容（口述、粘贴、转写）
- 需求/合同类文件

### 6.2 skill 产出的五步结果

`project-intake-pm` 会自动完成：判断类型、二进制转换、会议时间线更新、关键决策提炼、索引更新。产出物的存放位置：

| 产出 | 存放位置 |
|------|---------|
| 二进制原件 | 项目根 `_source/` 或 `01-项目基线/_source/` |
| 转换 MD | 与原件同目录，命名 `{原文件名}_转换.md` |
| 会议时间线条目 | `01-项目基线/会议记录/会议纪要_时间线.md`（倒序） |
| 会议原始纪要 | `01-项目基线/会议记录/原始纪要/` |
| 关键决策草拟 | `01-项目基线/关键决策.md`（#D0XX 编号，默认"待确认"） |
| 索引更新 | `00-项目总览.md`、`_index.md`、涉及子目录的 `_index.md` |

### 6.3 关键决策规则（人要记住的）

- 编号 `#D001` 起全局自增，不跳号、不复用
- 新决策默认状态"待确认"，必须人审阅后改"已确认"
- 后续文档引用决策用 `关联决策 #D003` 格式
- 被推翻的决策不删，标注"后续决策：#D005"

## 七、原型→规格→需求 回写规则

原型调整完成后，必须逐层回写：

1. **原型 → 规格**：原型中的功能/交互/状态变更，回写到对应规格文档
2. **规格 → 需求**：规格变更涉及需求描述变化的，回写到需求文档

**执行机制**：新版本完整复制原型后，跑 `strip-annotations.sh` 剥离旧标注得到干净骨架，在骨架上改动只对新改动处加标注。新标注天然就是回写清单——AI 读取标注列表即可知道本轮改了什么。定稿后按标注列表逐层回写。

工具脚本和标注规范详见 `prototype-html-pin` skill。

## 八、_index.md 维护规则

每个 `_index.md` 是 AI 进入该目录的"第一眼"，必须真实。状态失真会导致 AI 误判进度。

### 强制规则

- 版本阶段性完成后，运行 `sync-index.sh` 校正状态（由 `project-iteration` skill 收尾时自动执行）
- 不允许"索引标待启动"与"实际文件已存在"并存
- 新增/删除文件后，更新文件清单
- 每次评审或重大变更后，追加最近更新记录

### sync-index.sh 判定逻辑

脚本扫描版本目录，按文件存在情况自动判定状态：
- `02-需求文档/` 有 PRD → "需求完成"
- `03-规格文档/` 有规格 → "规格完成"
- `04-原型/` 有 HTML → "原型完成"
- `05-TAPD上传规划/` 有规划 → "TAPD完成"
- 全部完成 → 版本状态升级为"待收尾"

## 九、二进制文件转换

收到二进制文件（PDF/DOCX/XLSX/PPTX）时，转换流程由 `project-intake-pm` skill 自动执行。人只需要知道：

- 原件归档到 `_source/`，转换产物 `{原文件名}_转换.md` 同目录
- AI 读取时优先读 `_转换.md`，不读原始二进制
- 转换产物不完整时回退读原文

## 十、案例沉淀与知识库治理

### 10.1 案例沉淀标准

**触发条件**（满足任一，由 `project-retrospective-pm` skill 复盘时评估）：
- 项目阶段性完成，产出了可复用的设计模式或方案结构
- 项目中踩坑后提炼出值得记录的教训
- 复盘后发现某个决策逻辑可供后续项目参考

**案例格式**（至少包含）：背景、关键决策（选 A 而非 B 的理由）、踩坑记录、复用信号。

**领域分类**：按业务领域建立 `03知识库/工作案例库/` 子目录，一个领域积累 3 个以上案例后，提炼共性为方法论笔记（放 `01-通用规则/方法论/`）。

### 10.2 工作区级知识库维护规则

**`01-通用规则/` 各子目录**：

| 子目录 | 新增来源 | 更新时机 | 过期处理 |
|--------|---------|---------|---------|
| 模板/ | 项目中发现可复用结构 | 复盘或阶段性沉淀 | 旧版标记过期指向新版 |
| 检查清单/ | 评审中发现的漏检项 | 每次评审后补充 | 不再适用的条目删除 |
| 设计规范/ | 新类型项目的设计标准 | 项目启动时更新 | 版本化管理 |
| 设计库/ | 项目中沉淀的组件模式 | 原型设计完成后 | 保持精选不膨胀 |
| 方法论/ | 踩坑经验、可复用模式 | 积累 3 个相似案例后提炼 | 标注上下文限制 |
| 术语表/ | 项目中出现的行业术语 | 随时补充 | 定义变更同步更新引用 |

**`03知识库/` 各子目录**：

| 子目录 | 入库条件 | AI 读取规则 |
|--------|---------|------------|
| 工作案例库/ | 项目阶段性完成或有可复用结论 | 新项目启动时匹配领域 |
| 政策法规/ | 二进制需转换生成 `_转换.md` | 优先读转换产物 |
| 竞品分析/ | 调研完成即入库 | 作为 PRD 竞品章节输入 |
| 技术学习/ | 二进制需转换 | 需求设计时验证技术可行性 |
| 行业研究/ | 研究完成入库 | 立项和方案论证时引用 |

### 10.3 定期清理

每季度或按需执行一次：
1. 有效期检查（政策法规是否过期）
2. 转换时效检查（二进制更新后重新生成转换产物）
3. 案例库盘点（已完成项目是否沉淀）
4. 膨胀控制（过期内容标记或删除）
5. 交叉引用检查（过期内容的引用是否更新）

## 十一、AI 协作行为契约

以下规则已写入 `AGENTS.md`（工作区级，AI 自动加载），AI 每次执行相关任务时遵守：

1. 进入项目先读 `00-项目总览.md` 和 `02-迭代/_index.md`
2. 新建/修改需求前，先读 `01-项目基线/关键决策.md`
3. 开始原型设计前，先读对应规格文档
4. 原型迭代时，用户未声明新版本就在原页面改；声明新版本才走剥离流程
5. 收到二进制文件时，主动询问是否需要转换
6. 引用关键决策时，使用 `#D001` 编号格式
7. 交付前，自查是否标注决策来源和需求依据
8. 版本阶段性完成时，运行 `sync-index.sh` 校正状态
9. 版本收尾时，主动提醒复盘

### 能做什么

| 职责 | 说明 | 依赖 skill |

| 复盘驱动 | 系统复盘、识别缺陷、驱动修复 | project-retrospective-pm |
| 质量校验 | 执行文档生成纪律 10 条（见 AGENTS.md） | 内置规则 |

### 不能做什么

- **不做决策**：草拟关键决策默认"待确认"，技术选型、范围调整、商务判断最终决策权在 PM
- **不替代评审**：产出 PRD/原型后不自行判断"通过"，需 PM 正式评审
- **不跳过上下文**：不凭记忆工作，必须先读关键决策和会议时间线
- **不覆盖源文件**：处理二进制文件时，原始文件始终保留在 `_source/`
- **不主动建议新版本**：避免过度工程，用户说才建
