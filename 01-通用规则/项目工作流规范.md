# 项目工作流规范

_最后更新：2026-06-23 | 版本：V2.0（迭代型四层结构）_

本规范定义了工作区中每个项目从 0 到 1 的标准操作流程、文件关联方式和 AI 协作行为准则。

> **V2.0 变更要点**：从单层线性七目录升级为**迭代型四层结构**（项目 / 通用 / 迭代 / 版本内目录），整合实战验证有效的迭代约定。所有项目（含存量）统一采用本结构。

---

## 一、四层目录结构

```
项目名/
├── 00-项目总览.md              # 项目仪表盘（状态、迭代索引、关键决策索引）
│
├── 01-项目基线/                    # 【第一层】跨版本累积性资产
│   ├── 关键决策.md             # 跨阶段提炼的设计约束（#D001 自增编号）
│   ├── 需求池.md               # 所有候选需求的统一入口
│   ├── 业务规则/               # 结算规则、分润规则等永久约束
│   ├── 设计规范/               # design.md + 原型样式案例 + 品牌色/Logo
│   ├── 架构图/                 # 系统架构、脑图、页面流转图
│   ├── 法律协议/               # 合规约束
│   ├── 会议记录/               # 按时间组织的外部输入
│   └── 项目工具/               # 本项目的流程规范、模板
│
├── 02-迭代/                    # 【第二层】所有版本迭代的容器
│   ├── _index.md               # 迭代总索引（版本清单 + 状态）
│   ├── 01-V1.0.0/              # 【第三层】单个版本，完整自包含
│   │   ├── _index.md           # 版本索引（导航 + 状态 + 需求清单）
│   │   ├── 01-原始需求/        # 本版本的外部输入材料
│   │   ├── 02-需求文档/        # PRD + 增量需求文档 + 功能清单
│   │   ├── 03-规格文档/        # 原型设计规格（按页面编号）
│   │   ├── 04-原型/            # HTML 原型 + _scaffold/ 脚手架
│   │   ├── 05-TAPD上传规划/    # TAPD 需求上传规划
│   │   └── 06-变更记录/        # 本版本的需求变更清单、差异清单
│   ├── 02-V1.0.1/
│   └── ...
│
├── 03-对外材料/                # 汇报 PPT、产品介绍、白皮书
├── _source/                    # 所有二进制文件统一归档
│   └── *_转换.md               # 二进制→MD 转换产物，AI 优先读取
└── .workbuddy/memory/          # AI 记忆系统
```

> 注：原 V1.0 规范的 `01-项目输入` 已拆分——外部输入的累积性部分（关键决策、会议记录）并入 `01-项目基线/`，单版本的原始输入进 `版本/01-原始需求/`。原 `04-架构图`、`05-评审纪要`、`06-对外材料` 重组：架构图入通用，评审纪要并入变更记录，对外材料提升为顶层目录。

## 二、通用 vs 迭代的分界线（关键纪律）

这是四层结构的核心规则，决定一个文件该放哪：

| 01-项目基线/ | 02-迭代/版本号/ |
|---------|---------------|
| 跨版本累积性资产（关键决策、需求池、竞品分析、路线图） | 本版本专属产物 |
| 永久约束（品牌、法律协议、业务规则） | 需求文档、方案设计 |
| 系统级资产（架构图、脑图、页面流转图） | 规格文档、原型文件 |
| 按时间组织的资产（会议记录） | TAPD上传规划、变更记录 |
| 项目工具和流程规范 | 原始需求材料 |

**核心原则**：通用中的内容应是版本的**可靠输入**，不应包含未确定的实验性产物。迭代中产生的脑图、流程图、业务规则，先存当前版本的 `02-需求文档/` 或 `06-变更记录/`，**确定不再变动后再合并到 `01-项目基线/`**。

## 三、版本迭代规则

### 3.1 版本号与命名

- 版本文件夹：`0X-V{MAJOR}.{MINOR}.{PATCH}`，如 `01-V1.0.0`、`02-V1.0.1`
- 关键文件（需求文档、TAPD规划）文件名带版本号：`xxxV1.0.0.md`
- 原型文件和规格文件**不带版本号**，版本信息在路径上

### 3.2 版本资产（每个版本完整自包含）

每版本文件夹包含完整可交付成果：

| 目录 | 内容 |
|------|------|
| `01-原始需求/` | 本版本相关的外部输入材料 |
| `02-需求文档/` | PRD + 增量需求文档 + 功能清单 |
| `03-规格文档/` | 按页面编号的完整规格 |
| `04-原型/` | HTML 完整复制（非仅差异）+ `_scaffold/` 脚手架 |
| `05-TAPD上传规划/` | TAPD 需求上传规划 |
| `06-变更记录/` | 本版本的需求变更清单、差异清单、评审纪要 |

> **评审纪要处理**：不单设目录。评审结论作为变更条目记入 `06-变更记录/`，被评审文档标注修改来源。

**新版本从上一版本完整复制原型和规格，非仅复制差异。** 复制后跑 `strip-annotations.sh` 剥离旧标注，得到干净骨架再改（见第六章）。

## 四、工作流阶段

```
版本规划 → 需求撰写 → 规格+原型 → 评审 → TAPD上传 → 文档站发布 → 版本收尾+复盘
```

### 阶段 1：版本规划
1. 同步需求池（腾讯文档 → `01-项目基线/需求池.md`）
2. AI 读取需求池 + 关键决策，输出候选范围
3. 人确认 → 创建 `02-迭代/0X-VX.X.X/`
4. AI 生成 `06-变更记录/需求范围清单_VX.X.X.md`
5. 更新需求池状态列（候选→已纳入VX.X.X）

### 阶段 2：需求撰写
1. 参考 `01-项目基线/关键决策.md`、`01-项目基线/业务规则/`
2. 参考 `01-项目基线/架构图/` 脑图和流转图
3. 写入 `02-需求文档/`

**需求文档层次（精简原则）**：
- 默认只写**1份 PRD**（含背景/范围/字段/交互）
- 复杂需求**按需**追加方案设计文档，但默认不写，避免与 PRD 重复
- 每批次新需求出独立的**增量需求文档**（按主题命名），不用子版本号叠加
- 基线需求文档（如 MVP V1.0）内容冻结，变更以增量方式叠加

### 阶段 3：规格 + 原型
1. 参考 `01-项目基线/设计规范/` 取主色和样式
2. 参考 `01-项目基线/法律协议/` 检查合规
3. 写入 `03-规格文档/`，原型写入 `04-原型/`

**规格文档命名规范**：
- 新建页：`XX-页面名-原型设计规格.md`（完整规格）
- 变更页：`XX-页面名-变更规格.md`（仅变更描述，文件名后缀区分）
- 规格文档头部维护变更记录头（倒序），正文原地修改

**原型生成与迭代**：遵循 `prototype-html-pin` skill，含标注规范、脚手架、回写流程（见第六章）。

### 阶段 4：评审
- 评审结论作为变更条目记入 `06-变更记录/`
- 被评审文档标注修改来源：`修改依据：[[06-变更记录/xxx#评审项N]]`

### 阶段 5：TAPD 上传
- 生成 `05-TAPD上传规划/TAPD上传规划_VX.X.X_{端}.md`
- 按规划逐条上传

### 阶段 6：文档站发布
- 使用 `vitepress-deploy` skill 将原型同步到文档站
- **原型改完后必须同步**，不允许源项目与文档站脱节
- 部署脚本应包含自动同步步骤

### 阶段 7：版本收尾 + 复盘
1. 更新 `00-项目总览.md` 迭代状态
2. 更新 `02-迭代/_index.md` 版本索引
3. 运行 `sync-index.sh` 校正各版本 `_index.md` 状态（见第七章，由 project-iteration skill 牵引）
4. 上一版本的带标注原型**保留在原版本目录**，不额外归档；过程草稿如用户要求清理才移入 `归档/`
5. 输出汇报材料到 `03-对外材料/`
6. **执行复盘**（使用 `project-retrospective-pm` skill），覆盖：识别提示词缺陷、知识库缺失、可技能化的重复流程；更新 `01-项目基线/` 相关内容；将迭代中确定的脑图/规则合并到通用

## 五、原型→规格→需求 回写规则（核心，解决 W1）

原型调整完成后，**必须逐层回写**：

1. **原型 → 规格**：原型中的功能/交互/状态变更，回写到对应规格文档
2. **规格 → 需求**：规格变更涉及需求描述变化的，回写到需求文档

**执行机制（自动化，不再靠人工记忆）**：
- 新版本完整复制原型后，跑 `strip-annotations.sh` 剥离旧标注 → 得到干净骨架
- 在干净骨架上改动 → 只对新改动处加标注（编号从 1 计）
- **新标注天然就是回写清单**——AI 读取标注列表即可知道本轮改了什么，不用人工分辨
- 定稿后按标注列表逐层回写规格和需求

> 这是原 W1（"回写 skill 未落地"）的解决方案。详见第六章工具说明。

## 六、原型工具链（prototype-html-pin skill 配套）

### 6.1 公共样板脚手架

同系列页面 ≥5 个时启用（详见 skill）。在 `04-原型/_scaffold/` 维护公共 CSS/侧边栏/JS，源文件用占位符引用，`build.sh` 合并为单文件交付。

### 6.2 标注规范（剥离脚本的前提）

标注必须严格按以下结构生成，否则自动化工具无法识别：

| 要素 | 强制结构 |
|------|---------|
| 标注元素 | `<xxx class="proto-element" data-proto-id="数字">` |
| 说明条目 | `<div class="proto-desc doc-section" data-proto-desc="数字">` |
| 连线层 | `<svg id="connections-svg">` |
| 连线配置 | `window.connections = [...]` |

### 6.3 工具脚本

| 脚本 | 位置 | 作用 |
|------|------|------|
| `build.sh` | prototype-html-pin/assets | 合并脚手架占位符，生成单文件交付物（含自检） |
| `check-annotations.sh` | prototype-html-pin/assets | 扫描 HTML，报告标注结构是否规范 |
| `strip-annotations.sh` | prototype-html-pin/assets | 剥离标注/连线/右侧说明，产出干净骨架 |
| `screenshot-diff.sh` | prototype-html-pin/assets | 新旧 HTML 截图对比，验证视觉一致性 |
| `sync-index.sh` | project-iteration/assets | 扫描版本目录，自动校正 _index.md 状态列 |

### 6.4 迭代触发规则（重要）

- **用户未声明新版本** → 在原页面直接改，不剥离
- **用户声明"开始新版本迭代"** → 走完整复制 + 剥离 + 只标新改动流程
- AI 不主动建议"是否新建版本"，避免过度工程

## 七、_index.md 维护规则（解决状态失真）

每个版本目录的 `_index.md` 是 AI 进入版本的"第一眼"，**必须真实**。

### 7.1 强制规则

- 版本阶段性完成后，必须运行 `sync-index.sh` 校正状态
- 不允许"目录导航标待启动"与"实际文件已存在"并存
- 新增/删除文件后，更新文件清单
- 每次评审或重大变更后，追加最近更新记录

### 7.2 sync-index.sh 的判定逻辑

脚本扫描版本目录，按文件存在情况自动判定状态：
- `02-需求文档/` 有 PRD 文件 → "需求完成"
- `03-规格文档/` 有规格文件 → "规格完成"
- `04-原型/` 有 HTML 文件 → "原型完成"
- `05-TAPD上传规划/` 有规划文件 → "TAPD完成"
- 全部完成 → 版本状态升级为"待收尾"

## 八、二进制文件转换流程

### 触发条件
- 任何目录收到新的二进制文件（PDF/DOCX/XLSX/PPTX）
- AI 需要读取文件内容时

### 操作步骤
1. **归档原始文件**：二进制文件放入对应目录的 `_source/` 子目录
2. **按类型转换**：

| 源格式 | 转换工具 | 输出规范 |
|--------|---------|---------|
| PDF | `pypdf` 提取文本 | 逐页标注 `## 第N页`，保留原文结构 |
| DOCX | `python-docx` 提取段落 | 保留标题层级，正文为段落 |
| XLSX | `pandas` 提取表格 | 按 Sheet 分节，转 Markdown table |
| PPTX | `python-pptx` 提取文本 | 按页组织，标注标题和文本框 |

3. **命名**：转换产物为 `{原文件名}_转换.md`，与源文件同在 `_source/`
4. **AI 读取规则**：优先读 `_转换.md`，不读原始二进制；转换产物不完整时回退原文

## 九、AI 协作行为契约

以下规则写入 AGENTS.md（工作区级，ZCode 自动加载），AI 每次执行相关任务时遵守：

1. **进入项目先读 `00-项目总览.md` 和 `02-迭代/_index.md`**，了解当前版本状态
2. **新建/修改需求前**：先读 `01-项目基线/关键决策.md` 确认前置约束
3. **开始原型设计前**：先读对应规格文档，确认功能范围
4. **原型迭代时**：用户未声明新版本就在原页面改；声明新版本才走剥离流程
5. **收到二进制文件时**：主动询问是否需要转换
6. **引用关键决策时**：使用 `#D001` 编号格式
7. **交付前**：自查是否标注关键决策来源和需求依据
8. **版本阶段性完成时**：运行 `sync-index.sh` 校正 _index 状态
9. **版本收尾时**：主动提醒复盘，复盘按 `project-retrospective-pm` skill 执行

## 十一、文件关联速查表

| 从 | 到 | 关联方式 | 示例 |
|----|----|---------|------|
| 关键决策 | 需求文档 | 编号引用 | `决策依据：#D003` |
| 需求文档 | 原型规格 | 章节引用 | `需求依据：[[02-需求文档/xxx#3.2]]` |
| 原型规格 | HTML 原型 | 文件名对应 | `E12-上游通道-原型设计规格.md` ↔ `E12-上游通道-原型.html` |
| 变更记录 | 被评审文档 | 双向引用 | 纪要链文档，文档标注修改来源 |
| 二进制转换MD | 原始二进制 | 同目录+命名 | `_source/xxx.pdf` → `_source/xxx_转换.md` |
| 复盘报告 | Skill/规范 | 编号追溯 | `来源：XX项目复盘 YYYY-MM-DD` |

## 十一、跨工作区使用说明

本规范为工作区的主规范。副业等工作区通过绝对路径引用本文件，不另行维护副本：
- 副业工作区 AI 读取本规范时，以规则内容为准，忽略相对路径
- 副业工作区自有通用内容维护在本地 `01-项目基线/`
- 规范更新只需修改本文件，副业侧自动生效
