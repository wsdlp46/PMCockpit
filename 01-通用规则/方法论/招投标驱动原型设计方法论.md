# 招投标驱动原型设计方法论

> 从招投标文档到可交互原型的完整流水线。修改大模型指令即可执行。

---

## 一、适用场景判定

### 启动条件（满足任一即适用）

1. 需求来源为招投标文件/合同附件/Excel功能清单，而非用户调研
2. 功能边界由外部文档约定，不可自行扩展
3. B端管理后台/门户类产品，需按端-模块-页面三级展开

### 不适用

- C端产品（角色单一，无需端侧拆分）
- 自研产品（需求来自用户/市场，边界可自主调整）
- 纯工具型产品（功能少，无需分层流程）

---

## 二、六阶段流水线

```
招投标文档 → [S1 文档解析] → 功能清单
                                    ↓
                [S2 架构定义] → 完整产品需求文档（定位/用户/流程/功能/页面/规则）
                                    ↓
                [S3 规格生成] → 逐页设计规格MD
                                    ↓
                [S4 原型生成] → 逐页HTML原型
                                    ↓
                [S5 复盘归因] → 问题清单 + 归因表
                                    ↓
                [S6 结晶沉淀] → 更新 Skill/项目基线/术语
```

**每个阶段有判定门，未通过不进入下一阶段。**

---

## S1：文档解析

### 输入

- 招投标Word终稿（.docx）
- Excel功能清单（.xlsx）

### 执行步骤

1. **读取Word**：提取项目名称、采购人、预算、技术要求章节
2. **读取Excel**：按"二级模块-三级功能"提取全部功能条目
3. **交叉比对**：逐项比对Word文字描述 vs Excel功能清单
   - 在Word中但不在Excel → 标记"文字提及但无清单对应"
   - 在Excel中但Word未展开 → 标记"清单有但无文字描述"
4. **输出功能清单表**：

| 编号 | 二级模块 | 三级功能 | 功能描述（原文） | Word提及 | Excel对应 | 备注 |

### 判定门

- [ ] 功能清单覆盖Excel全部条目（无遗漏）
- [ ] 交叉比对的差异已标记
- [ ] 用户确认"严格以Excel为准"原则

### 输出

- 结构化功能清单表（写入项目 `03-功能清单/` 目录）
- 差异标注清单（供S2决策参考）

### 自动化程度

**高**。大模型可直接读取docx/xlsx，提取和比对无需人工介入。用户仅需确认"以哪个文档为准"。

---

## S2：架构定义

### 输入

- S1产出的功能清单
- 项目基线（01-项目基线/，读取历史架构和术语）

### 执行步骤

#### S2.1 端侧模型定义

1. 读取项目基线（01-项目基线/架构图/、_index.md）是否有历史端侧模型
2. 如有，基于历史模型给出推荐方案，附对比表：

| 端 | 历史模型中的角色 | 本模块是否需要 | 理由 |

3. 如无，从功能清单中提取所有操作主体，归纳端和角色
4. 输出端侧模型图（端-角色-功能数量统计）
5. **等待用户确认**，不可跳过

#### S2.2 模块划分与页面映射

1. 将功能清单按二级模块分组
2. 每个二级模块下，按业务流程确定页面数量
3. 每页标注承载的功能编号
4. 识别业务闭环缺口（功能清单未覆盖但流程必须的环节）

**闭环缺口识别规则**：
- 有"创建"无"删除/作废" → 缺逆向操作
- 有"提交"无"审批/审核" → 缺审批环节
- 有"申请"无"结果通知" → 缺反馈环节
- 有"配置"无"生效/停用" → 缺状态管理
- 闭环缺口需标注"补"字，与原清单功能区分

5. 输出功能架构与页面清单：

| 端 | 模块 | 页面编号 | 页面名称 | 承载功能编号 | 闭环补缺 |

#### S2.3 已有能力确认

逐一询问用户：
- 平台是否有现成流程引擎？→ 有则砍掉审批类独立页面
- 平台是否有现成消息/通知组件？→ 有则不重复设计
- 平台是否有统一认证/用户体系？→ 有则不独立设计登录注册
- 是否有已有子系统可复用？→ 有则标注复用关系

#### S2.4 业务流程确认

1. 画出核心业务流程图（申请→审批→开通→使用→评价等）
2. 在流程图上标注每页的位置和上下游
3. **等待用户确认**，不可跳过

### 判定门

- [ ] 端侧模型已确认（用户明确说了"可以"）
- [ ] 每个功能已分配到端和页面（无遗漏无悬空）
- [ ] 闭环缺口已标注并经用户确认
- [ ] 已有能力已确认（避免重复设计）
- [ ] 业务流程图已确认

### 输出

- **完整产品需求文档**（写入项目 `02-需求文档/` 目录），包含以下7个板块：
  1. **产品定位**——一句话定义、建设范围、功能数量统计、边界（"不是什么"）
  2. **信息架构与用户分析**——端侧模型 + 角色定义（每个角色"一句话定义"+典型用户画像）+ 功能项数分配
  3. **核心业务流程**——端到端业务闭环 + 分流程段展开 + 每个节点标注对应页面
  4. **使用场景（带角色）**——5-6个用户故事级别场景，每个带角色和完整操作路径
  5. **功能清单 + 模块映射**——按模块分组，每行标注招投标功能编号（流水线的"追溯码"）
  6. **功能架构与页面清单**——端→模块→页面→功能四级映射表，标注闭环新增项（写入项目 `03-功能清单/` 目录）
  7. **业务规则**——Out of Scope + 设计约束 + 待决事项（含影响范围和原型状态）+ 验收标准
- 端侧模型图
- 业务流程图

### 自动化程度

**中**。端侧推荐和页面映射可自动生成，但4项确认必须用户介入。预期5轮对齐减到2-3轮。

## S3：规格生成

> **为什么S3不能跳过？** S2已经有了页面清单和功能编号，有人会想直接让AI生成HTML原型。但S3的规格MD是S4的**必要输入**——没有规格文档，prototype-html不知道每个页面该有哪些字段、什么交互逻辑、弹窗层级是L1还是L2，只能靠AI自由发挥，而自由发挥=翻车。S3是施工图，S4是施工现场。没有施工图，施工就是赌博。

### 输入

- S2产出的页面清单
- S1产出的功能清单（原文描述）
- 项目基线（01-项目基线/，术语和约定）
- `01-通用规则/方法论/原型设计规范.md`

### 执行步骤

#### S3.0 准备检查

1. 确认设计规范文件存在：`01-通用规则/设计规范/Web应用设计规范.md`
   - 不存在 → 提醒用户"缺少设计规范，原型生成阶段将无法自动适配视觉参数"
2. 确认项目基线中术语是否已记录（01-项目基线/ 或 01-通用规则/术语表.md）
   - 没有 → 提醒用户"后续遇到业务术语将逐条确认"

#### S3.1 逐页生成（循环）

对页面清单中的每个页面，执行以下流程：

**步骤A：读取功能原文**

读取该页面对应的功能编号在功能清单中的原文描述。不得凭记忆或推断编写。

**步骤B：输出规格文档**

按以下固定结构输出，每页一个独立MD文件：

```
# [编号] [页面名称] — 原型设计规格

> 端：[X] | 模块：[X] | 操作角色：[X]
> 承载功能：[编号和名称]

## 零、功能概述
  1. 核心业务（2-3句）
  2. 关键概念（数据模型/状态流转）
  3. 与上下游关系（输入来源/输出去向）

## 一、权限说明
  1.1 数据权限
  1.2 功能权限（表格：序号/功能权限/说明）

## 二、页面结构
  一句话说明页面组件组成

## 三至末尾、页面内容
  按区域逐级展开，使用标准表格类型

## 末、交互规则
  仅放全局约束，不重复已有内容
```

**步骤C：状态联动三表（有状态的页面必做）**

当页面存在多状态管理时，必须同时输出三个表：

| 表 | 内容 | 用途 |
|---|------|------|
| 状态标签表 | 序号/状态/颜色/说明 | 定义有哪些状态 |
| 状态与操作对应表 | 序号/状态/可用操作 | 定义每个状态能做什么 |
| 操作行为说明表 | 序号/操作/弹窗层级/行为 | 定义每个操作的具体行为 |

三表必须一一对应，不允许出现"状态表有但操作表缺"的情况。

**步骤D：等待用户确认**

输出后暂停，等待用户明确说"可以""继续""OK"后才进入下一页。
用户提出修改 → 修改后再确认 → 不跳过。

#### S3.2 模板进化

前2页（第一批）为模板建立期，结构允许不完美。
从第3页起，模板应趋于稳定。如出现新的表格类型或章节需求，需回溯更新已出页面。

### 关键约束（铁律）

| # | 约束 | 说明 | 来源教训 |
|---|------|------|---------|
| C1 | 不自创功能 | 所有功能描述严格来源于功能清单，新增闭环功能需标注"补"并单独说明。**操作按钮也属于功能**（如"导出""批量删除"），必须在功能清单中有对应，不可自行添加 | 某项目V2.0写了9个非Excel子系统；后续项目发现AI每页自动加"导出"按钮但清单无此功能 |
| C2 | 不自加字段 | 表格字段仅来自功能描述原文，新增字段需与用户确认 | M3自创"所属服务"字段 |
| C3 | 不跳确认 | 每页输出后必须等用户确认 | — |
| C4 | 命名对齐原文 | 模块名/字段名/按钮名/状态名必须与招投标原文或用户确认的术语一致 | "新增API→定义API""废弃→下架" |
| C5 | 交互复杂度控制 | 优先简单交互（行列按钮/开关/下拉），拖拽/树形编辑等需确认 | M3拖拽排序被否 |
| C6 | 弹窗层级规范 | L0(列表) / L1(侧滑z-1000) / L2(弹窗z-1050) / L3(确认z-1100)，同层最多1个 | 标注系统3次弹窗穿透崩溃 |

### 判定门

- [ ] 全部页面规格已输出并经用户确认
- [ ] 每个功能编号在至少一个页面中被承载
- [ ] 闭环缺口全部在规格中有对应设计
- [ ] 术语使用与用户确认的一致（无自造词）

### 输出

- N个原型设计规格MD文件（每页一个，写入 `03-功能清单/` 或 `04-原型图/` 目录）

### 自动化程度

**高**。基于proto-spec-generator Skill可自动生成，用户仅需逐页确认和修正。Skill固化率70%+。

## S4：原型生成

### 输入

- S3产出的规格MD文档
- `01-通用规则/设计规范/Web应用设计规范.md`（视觉参数）

### 执行步骤

#### S4.1 风格判定

读取设计规范，判定页面类型：

| 页面类型 | 布局方案 | 典型特征 |
|---------|---------|---------|
| 管理后台/B端 | 左侧导航+顶栏+内容区 | 表格/表单/弹窗为主 |
| 门户首页/C端 | 顶栏+卡片式内容 | 营销/展示/引导为主 |
| 数据大屏 | 全屏深色背景+图表 | 数据可视化为主 |
| 移动端H5 | 底部Tab+滚动流 | 触控交互为主 |

#### S4.2 规格匹配（强制步骤，不可跳过）

生成前必须先列出将要包含的UI元素清单：
- 从规格文档中提取所有字段、按钮、弹窗、状态
- 逐项核对：规格要求的 → 生成；规格未提及的 → 不生成
- 输出核对清单，用户可快速扫一眼

**规格匹配铁律**：多一个字段就是错，少一个功能就是漏。

#### S4.3 逐页生成

1. 读取规格MD，按规格匹配清单生成HTML
2. 视觉参数从设计规范MD读取，不硬编码
3. 每页生成后运行交付前自查清单

#### S4.4 交付前自查（强制步骤）

| 检查项 | 验证方式 |
|--------|---------|
| 字段一致性 | 规格中每个字段在HTML中有对应元素 |
| 交互完整性 | 已操作过的能否撤销/修改/删除（三问） |
| 组件完整性 | 规格中每个按钮/弹窗在HTML中存在 |
| 业务规则 | 规格中的状态联动在HTML中正确实现 |
| 术语一致性 | 按钮名/列名/状态名与规格一致 |

#### S4.5 交付标注（强制步骤）

每个HTML原型页面顶部必须添加灰色小字标注：

> **"仅供交互演示，非最终代码。字段值为占位数据，交互逻辑为简化演示。"**

**为什么必须加**：HTML原型做得越逼真，研发越容易误以为是最终代码。这条标注的研发阅读率，比任何口头提醒都高。

#### S4.6 多页面系列规则（页面数>1时必做）

1. 所有页面共享同一侧边栏结构
2. 侧边栏菜单项使用 `<a href="文件名.html">` 实现导航
3. 当前页面菜单项标记 active 态
4. 新页面创建后，自动更新同系列所有已有页面的侧边栏
5. 页面创建顺序无关，但侧边栏必须保持一致

#### S4.6 等待用户确认

同S3，逐页确认。修正后重新确认。

### 判定门

- [ ] 全部页面HTML已生成并经用户确认
- [ ] 自查清单全部通过
- [ ] 交付标注已添加（每个页面顶部）
- [ ] 多页面导航联通（如适用）
- [ ] 侧边栏一致（如适用）

### 输出

- N个交互式HTML原型文件（写入 `04-原型图/` 目录）

### 自动化程度

**高**。基于prototype-html Skill可自动生成，用户仅需逐页确认和修正。

---

## S5：复盘归因

### 输入

- S3和S4过程中记录的所有用户修正

### 执行步骤

#### S5.1 问题收集

整理用户修正记录，逐条列出：
- 问题描述
- 发生场景（哪个页面/哪个字段/哪个交互）
- 严重度（高/中/低）

#### S5.2 归因分类

将每个问题归入以下三类之一：

| 归因 | 定义 | 修复位置 |
|------|------|---------|
| 提示词缺失 | Skill/指令中没有约束该行为 | 更新Skill |
| 知识库缺失 | 项目特有的业务规则/术语未记录 | 补充到 01-项目基线/（业务规则/、关键决策.md） |
| 流程缺失 | 工作流中缺少必要步骤/检查 | 更新方法论 |

#### S5.3 输出归因表

| # | 类型 | 问题描述 | 发生场景 | 严重度 | 归因 | 修复位置 |

### 判定门

- [ ] 所有问题已归因
- [ ] 每个归因有明确的修复位置

### 输出

- 归因表（写入复盘报告）

### 自动化程度

**中**。问题收集可自动（从对话中提取修正记录），归因判断需人工审核。

---

## S6：结晶沉淀

### 输入

- S5的归因表

### 执行步骤

#### S6.1 按归因类型修复

| 归因类型 | 执行动作 | 写入位置 |
|---------|---------|---------|
| 提示词缺失 | 将约束规则写入Skill | `.ai/skills/` |
| 知识库缺失 | 将业务术语写入术语表，规则写入项目基线 | `01-项目基线/（业务规则/、需求池.md）` |
| 流程缺失 | 将新步骤写入方法论 | `01-通用规则/方法论/招投标驱动原型设计方法论.md` |

#### S6.2 知识分层检查

检查每个知识点是否放在了正确的位置：

| 知识类型 | 应存放位置 | 不应存放位置 |
|---------|-----------|------------|
| 通用流程框架（"先确认流程再逐页"） | Skill | — |
| 项目特有业务规则（"包月=30天"） | 项目基线 01-项目基线/业务规则/ | Skill（破坏通用性） |
| 视觉参数（品牌色/字号/圆角） | `01-通用规则/设计规范/` | Skill（硬编码无法跨项目） |
| 跨项目方法论（本文件） | `01-通用规则/方法论/` | Skill（Skill是执行者不是存储者） |
| 决策记录（为什么选A不选B） | 01-项目基线/关键决策.md | — |

#### S6.3 术语表更新

将项目中出现的新术语记录下来（通用术语进 01-通用规则/术语表.md，项目专有术语进 01-项目基线/）：

| 术语 | 定义 |

#### S6.4 案例更新

更新案例库文档，将本次经验沉淀：
- 新增的关键决策点
- 新增的失败尝试
- 新增的准则/约束

### 判定门

- [ ] 所有归因项已修复
- [ ] 知识分层检查通过（无错位）
- [ ] 术语表已更新
- [ ] 案例文档已更新（如有重要经验）

### 输出

- 更新后的Skill文件
- 更新后的项目基线文件（关键决策.md、业务规则/、需求池.md）
- 更新后的方法论文件
- 更新后的案例文档（如有）

### 自动化程度

**中低**。写入动作可自动，但知识归层判断需人工审核。建议"自动写入草稿 → 用户确认 → 正式更新"。

## 三、决策树

### 遇到"功能清单没写但业务流程需要"时

```
功能清单无 → 业务流程必须？ → 是 → 标注"补"，写入页面清单
                              → 否 → 不做，列入Out of Scope
```

### 遇到"不确定术语该叫什么"时

```
术语不确定 → 招投标原文有？ → 有 → 严格用原文
                            → 无 → 用户确认 → 写入术语表
                                   → 用户未确认 → 标注"待确认"，不自行命名
```

### 遇到"已有能力可能可复用"时

```
可能有现成能力 → S2阶段已确认？ → 已确认 → 按确认结果处理
                                   → 未确认 → 暂停，先问用户
```

### 遇到"规格和原型之间发现矛盾"时

```
发现矛盾 → 原因是什么？ → 规格写错了 → 修改规格 → 重新生成原型
                          → 原型理解错了 → 修改原型 → 重新自查
                          → 业务逻辑有变 → 回到S2重新对齐 → 修改规格 → 修改原型
```

### 遇到"项目是第二个模块，之前做过一个模块"时

```
第二个模块 → 检查已有沉淀 → 有项目基线 → 读取术语和约定，跳过重复对齐
                               → 有Skill → 按Skill流程执行
                               → 有设计规范 → 自动适配视觉参数
                               → 三者都有 → S2可半自动，S3/S4高度自动
                               → 三者都没有 → 从S1开始完整执行
```

## 四、回退机制

| 当前阶段 | 触发回退的条件 | 回退到 |
|---------|-------------|--------|
| S3 规格生成 | 发现端侧模型错误（功能归属到错误的端） | S2 重新定义端侧 |
| S3 规格生成 | 发现业务流程理解偏差（页面间流转断裂） | S2 重新确认流程 |
| S4 原型生成 | 发现规格文档有重大遗漏/错误 | S3 修改规格后再生成 |
| S4 原型生成 | 发现设计规范缺失/不适用 | 暂停，先补充设计规范 |
| S5 复盘 | 归因发现流程性缺陷 | S6 修复方法论，下个模块生效 |
| 任意阶段 | 用户说"方向不对""重来" | S2 重新对齐 |

**回退原则**：宁早勿晚。越早发现问题，回退成本越低。

## 五、三层知识架构

> 核心方法论：Skill 存流程框架、项目基线存业务知识、设计规范存视觉参数。

| 层 | 存储位置 | 存什么 | 不存什么 | 生命周期 |
|---|---------|--------|---------|---------|
| 流程层 | `.ai/skills/` | 通用执行步骤、判定门、约束规则、表格模板 | 项目特有术语、业务规则、配色参数 | 跨项目持久 |
| 业务层 | 项目 `01-项目基线/（关键决策.md、需求池.md、业务规则/）` | 术语定义、约定规则、决策记录、待决事项 | 通用方法论、视觉参数 | 项目内持久 |
| 视觉层 | `01-通用规则/设计规范/` | 品牌色/字号/圆角/间距/组件规格 | 业务规则、执行流程 | 跨项目持久，按项目可覆盖 |

**错位的后果**：
- 业务知识写进Skill → 换项目时Skill带上了上个项目的"包月=30天"，破坏通用性
- 视觉参数硬编码在Skill → 换品牌色要改Skill，不如读设计规范文件
- 流程步骤写在项目基线 → 换项目重做，因为项目基线不跨项目

## 六、首次项目 vs 后续项目

| 步骤 | 首次项目 | 第二个项目起 |
|------|---------|------------|
| S1 文档解析 | 完整执行 | 完整执行（每个项目文档不同） |
| S2 架构定义 | 5轮对齐 | 2-3轮确认（读取历史架构推荐） |
| S2.3 已有能力 | 逐项询问 | 读取项目基线自动跳过已确认项 |
| S3 规格生成 | 模板从零建立 | 模板已有，直接用 |
| S3 准备检查 | 可能缺少设计规范 | 设计规范已有 |
| S4 原型生成 | 风格判定+视觉参数首次适配 | 自动读取设计规范 |
| S5 复盘 | 问题多（12个+） | 问题少（3-5个） |
| S6 结晶 | 大量新知识沉淀 | 增量更新 |

**预期时间**：首次项目 8-10天 → 后续项目 3-4天

## 七、嵌入Skill的指令格式

以下指令可直接嵌入proto-spec-generator或prototype-html Skill：

```
## 前置检查
1. 读取项目 01-项目基线/关键决策.md 和 业务规则/ → 获取项目约定
2. 读取 01-通用规则/设计规范/Web应用设计规范.md → 获取视觉参数
3. 两份文件都存在 → 继续；任一缺失 → 提醒用户

## 执行约束
- 不自创功能：所有功能描述严格来源于功能清单（操作按钮也属于功能，必须在清单中有对应）
- 不自加字段：表格字段仅来自功能描述原文
- 不跳确认：每页输出后必须等用户确认
- 命名对齐原文：术语必须与招投标原文或项目基线记录的术语一致
- 交互复杂度控制：优先简单交互
- 弹窗层级：L0/L1/L2/L3，同层最多1个

## 完成后动作
- 将新发现的术语记录到项目基线（通用术语进术语表，项目专有进 01-项目基线/）
- 将决策记录追加到 01-项目基线/关键决策.md（#D0XX 编号）
- 将方法论层面的改进写入 01-通用规则/方法论/招投标驱动原型设计方法论.md
```
