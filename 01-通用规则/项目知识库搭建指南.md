# 项目知识库搭建指南（含大知识库治理）

> 覆盖项目级知识库（单项目）和 Workspace 级大知识库（跨项目共享）的完整搭建与治理规范。按本文配置后，AI 可自动完成材料入库、关键决策提炼、索引维护、案例沉淀和文档质量控制。

---

## 第一部分：体系总览

## 一、双层知识库是什么

本工作流包含两层知识库，分工不同：

```
大知识库（Workspace 级，跨项目共享）
  工作区根/
  ├── 01-通用规则/          ← 模板、检查清单、设计规范、方法论
  ├── 03知识库/        ← 案例库、政策法规、竞品分析、技术学习
  └── 00主页.md        ← 工作区入口索引

        │ 引用（项目引用大知识库的模板/案例/规范）
        ▼

项目知识库（Project 级，单项目专用）
  项目名/
  ├── 00-项目总览.md   ← 项目仪表盘（一句话+关键事实+当前阶段）
  ├── 01-项目输入/     ← 本项目的会议时间线、关键决策
  ├── 02-需求文档/     ← 本项目的 PRD
  ├── ...
  └── .workbuddy/memory/ ← 本项目的 AI 记忆
```

| 层级 | 内容 | 共享范围 | 生命周期 |
|------|------|---------|---------|
| **大知识库** | 通用模板、设计规范、案例库、政策法规、竞品分析 | 所有项目共享 | 持续累积，不随项目结束而归档 |
| **项目知识库** | 本项目输入、需求、原型、评审、对外材料 | 本项目内 | 随项目启动创建，结束后可沉淀案例到大知识库 |

### 大知识库（Workspace 级）目录结构

```
01-通用规则/
├── _index.md             # 统一入口索引
├── 项目工作流规范.md       # 节点操作规则 + AI行为契约（操作手册）
├── 项目知识库搭建指南.md   # 双层体系与治理规范（本文件）
├── WorkBuddy智能体提示词.md # WorkBuddy智能体创建提示词
├── 术语表.md              # 行业术语定义统一词典
├── 模板/                  # PRD模板、方案模板、会议模板、上线模板
├── 检查清单/              # 需求评审检查清单、设计评审检查清单
├── 设计规范/              # 引用自外部的固定设计标准（不可修改参数）
├── 设计库/                # 原型基线 + 跨项目设计约定 + 设计约定.md
└── 方法论/                # 模式笔记、工作方法论

03知识库/
├── _index.md              # 目录索引 + 已有资料清单
├── 工作案例库/             # 按领域分类的历史项目案例
├── 政策法规/               # 行业标准、政策文件
├── 竞品分析/               # 竞品调研文档
├── 技术学习/               # 技术资料、学习笔记
├── 行业研究/               # 行业洞察、趋势分析
└── 读书笔记/               # 书籍阅读笔记
```

### 项目知识库（Project 级）目录结构

```
项目名/
├── 00-项目总览.md           # 项目仪表盘（一句话+关键事实+当前阶段+背景）
├── _index.md                # 项目文件全量索引 + 最近更新日志
├── 01-项目输入/             # 外部输入（合同/招标/会议记录/需求变更）
│   ├── _index.md
│   ├── 会议纪要_时间线.md    # 按时间记录每次外部输入及影响
│   ├── 会议纪要/             # MD格式会议原始记录
│   │   └── 原始纪要/         # 原始未精炼的会议文档
│   ├── 关键决策.md           # 跨阶段提炼的关键设计约束（#D001起）
├── 02-需求文档/             # PRD / 需求分析
│   └── _index.md
├── 03-原型设计/             # 设计规格 + 功能清单 + HTML 原型
│   └── _index.md
├── 04-架构图/               # 系统 / 业务 / 数据架构
│   └── _index.md
├── 05-评审纪要/             # 需求评审 / 设计评审 / 原型评审
│   └── _index.md
├── 06-对外材料/             # 汇报 PPT / 产品介绍 / 白皮书
│   └── _index.md
├── _source/                 # 所有二进制文件统一归档
│   └── *_转换.md            # 二进制→MD 转换产物，AI 优先读取
└── .workbuddy/
    └── memory/              # AI 记忆系统
```

操作细节（节点规则、转换工具、读取优先级的完整步骤）详见 `项目工作流规范.md`。

## 二、两层的关系与交互

### 2.1 启动阶段：从大知识库拉取

| 场景 | 拉取内容 | 存放位置 |
|------|---------|---------|
| 项目启动 | 复制 PRD 模板、方案模板 | 项目 02-需求文档/ |
| 需求撰写 | 引用术语表定义、政策法规标准 | 项目文档内链接引用 |
| 原型设计 | 引用设计规范、设计库组件 | 原型规格中标注规范依据 |
| 评审准备 | 使用检查清单 | 05-评审纪要/ 中引用 |
| 竞品分析 | 查阅竞品分析/ 已有报告 | 需求文档中引用结论 |

项目文档中通过双链引用大知识库内容：
- 引用政策法规：`[[../../03知识库/政策法规/xxx_转换#章节]]`
- 引用设计规范：`[[../../01-通用规则/设计规范/xxx]]`
- 引用案例：`[[../../03知识库/工作案例库/02-数据空间/案例_xxx]]`

### 2.2 结束阶段：反向沉淀到大知识库

| 沉淀内容 | 目标位置 | 触发条件 |
|---------|---------|---------|
| 可复用设计模式 | 01-通用规则/设计库/ | 项目中验证有效且≥2次复用 |
| 新踩坑经验 | 01-通用规则/方法论/踩坑笔记 | 复盘后判断有通用价值 |
| 完整项目案例 | 03知识库/工作案例库/ | 项目复盘后执行 |
| 新术语 | 01-通用规则/术语表/ | 项目中首次定义且跨项目通用 |
| 新检查项 | 01-通用规则/检查清单/ | 评审中发现既往清单未覆盖 |

## 三、索引体系

每个目录必须有一个 `_index.md`，作为 AI 进入该目录的"第一眼"。项目顶层额外配置 `00-项目总览.md` 作为仪表盘。

### 顶层 `00-项目总览.md` 模板（仪表盘）

```markdown
# 项目名 — 项目总览

_最后更新：YYYY-MM-DD_

## 快速入口
- 需求文档 — 原型设计 — 架构图 — 评审纪要 — 对外材料

## 一句话简介
> 一句话描述项目

## 当前阶段
[启动 / 需求 / 设计 / 开发 / 验收 / 已交付]

## 关键事实
- 客户/甲方：XXX
- 预算/规模：XXX
- 核心约束：XXX

## 项目结构
[目录树，标注各目录用途]

## 最近更新
- YYYY-MM-DD | 变动内容 | 关联文件
```

### 顶层 `_index.md` 模板（文件索引）

```markdown
# 项目名 — 文件索引

> 一句话描述项目

## 目录导航

| 目录 | 文件数 | 说明 |
|------|--------|------|
| 01-项目输入 | N | 含 _source/ 原始记录 |
| 02-需求文档 | N | ... |
| ... | ... | ... |

## 最近更新

- YYYY-MM-DD | 变动内容 | 关联文件
```

### 子目录 _index.md 模板

```markdown
# 目录名 — 目录索引

> 一句话描述

## 文件清单

| 文件 | 说明 | 最后更新 |
|------|------|----------|
| `xxx.md` | 说明 | YYYY-MM-DD |
```

### 维护规则

- 新增或删除文件后，必须更新对应目录的 `_index.md` 和顶层 `_index.md`
- 顶层 `_index.md` 的"最近更新"追加新条目
- 项目状态变更后，更新顶层 `00-项目总览.md`

## 五、材料入库与决策提炼

> **驱动 Skill**：`project-intake-pm`

每次收到新文件或会议内容时执行，由上述 Skill 自动完成五步流程。

### 5.1 步骤 1：判断类型

| 类型 | 识别信号 | 动作 |
|------|---------|------|
| PDF/DOCX/XLSX/PPTX | 扩展名 | 归档 _source/ → 执行转换（步骤 2） |
| 会议记录 | 口述/粘贴内容，或文件名含"会议""沟通""纪要" | 提取时间/来源/内容（步骤 3） |
| 需求/合同类 | 文件名含"合同""需求""招标""SOW" | 存放 01-项目输入/，标记为原始输入 |

### 5.2 步骤 2：二进制转换

- 所有二进制文件（PDF/DOCX/XLSX/PPTX）统一归档至项目根 `_source/`（不存在则新建）
- 转换产物 `{原文件名}_转换.md` 与源文件同放在 `_source/` 中
- 转换工具及优先级判定详见 `项目工作流规范.md` 第四章
- AI 读取时优先读 `_source/` 中的 `_转换.md` 产物，不读原始二进制

### 5.3 步骤 3：会议时间线更新

追加到 `01-项目输入/会议纪要_时间线.md`：

```markdown
## YYYY-MM-DD | [参会人员] [会议主题]

**核心内容**：（3句话概括）

**对项目的影响**：
- 影响1：具体描述
- 影响2：具体描述

**关联**：[[原始记录路径]]
```

**写入规则**：按时间倒序排列，关联字段必须指向具体原始文件路径。会议原始记录存放 `01-项目输入/会议纪要/原始纪要/`。

### 5.4 步骤 4：关键决策提炼

扫描决策信号词（"确定""决定""不做了""砍掉""甲方说""客户要求"），草拟条目：

```markdown
### #DXXX | YYYY-MM-DD | [决策标题]

**依据**：（为什么要这样决定）
**影响范围**：（功能范围/技术选型/交互模式/报价/排期）
**关联输入**：[[会议纪要_时间线#YYYY-MM-DD]]
**状态**：待确认
```

**关键规则**：
- 编号 #D001 起全局自增，读取已有最大编号再 +1
- 新决策默认状态为"待确认"，禁止自动写入"已确认"——必须等用户审阅后手动修改
- 后续需求文档中引用决策时使用 `关联决策 #D003` 格式
- 每条决策"关联输入"必须指向对应会议时间线条目，如 `[[会议纪要_时间线#2026-06-17]]`
- 引用其他决策时用 `[[#D007 标题]]`
- 后续被推翻的决策在"后续决策"字段记录：如 `后续决策：#D005`

### 5.5 步骤 5：更新索引

- 更新项目仪表盘 `00-项目总览.md` 的"最近更新"
- 顶层 `_index.md` 文件清单更新、"最近更新"追加新条目
- 涉及子目录的，同步更新子目录 `_index.md`

### 入库完成输出规范

```markdown
[项目名] 入库完成

文件：
- [文件名] → [存放位置]

提炼：
- 会议时间线新增 X 条
- 关键决策草拟 #Dxxx（待确认）

影响提醒：
- [如有影响现有需求的条目，在此标注]
```

## 七、AI 上下文加载与记忆系统

### 7.1 上下文加载规则

> **驱动规则**：Agent 内置上下文加载步骤（见 Agent 定义 `## 上下文加载` 节）

AI 接到任务后，按以下顺序加载上下文（不得跳过）：

| 步骤 | 内容 | 层级 | 说明 |
|------|------|------|------|
| 1 | 读取项目 `00-项目总览.md` 和 `_index.md` | 项目级 | 了解当前项目全貌和文件清单 |
| 2 | 读取 `01-项目基线/关键决策.md` | 项目级 | 确认本项目的所有前置设计约束 |
| 3 | 读取 `01-项目基线/会议记录/` 下的最新文档 | 项目级 | 了解本项目最新外部输入 |
| 4 | 读取 `01-通用规则/项目工作流规范.md` | 大知识库 | 确认跨项目的通用节点操作规则 |
| 5 | 打开案例库主页 | 大知识库 | 准备匹配同领域历史案例 |
| 6 | 读取 `03知识库/知识库主页.md` | 大知识库 | 查阅相关标准、法规、竞品分析 |

### 7.2 记忆系统

**长期记忆（.workbuddy/memory/MEMORY.md）**

记录项目级长期约定：
- 设计约定（颜色规范、单位体系、组件规范）
- 原型文件结构
- 案例库位置和模板

格式要点：按主题分组，具体数值写清楚（如颜色 HEX），触发词写明确（如"触发词：整理上线计划"）。

**工作日志（.workbuddy/memory/YYYY-MM-DD.md）**

每次完成实质性工作后追加：做了什么、关键决策或变更、注意事项。

**维护规则**：
- 每日日志只追加不覆盖
- 超过 30 天的日志提炼到 MEMORY.md，删除旧日志文件
- 不记录临时路径、搜索结果、工具错误等瞬时信息

## 十、案例库沉淀与定期清理

### 10.1 案例库沉淀标准

**触发条件**（满足任一即可）：
- 项目阶段性完成，产出了可复用的设计模式或方案结构
- 项目中踩坑后提炼出值得记录的教训
- 复盘后发现某个决策逻辑可供后续项目参考

**案例格式**（至少包含）：
- 背景（什么项目/场景触发）
- 关键决策（选了 A 而非 B，为什么）
- 踩坑记录（怎么发现的，怎么解决的）
- 复用信号（什么情况下可以再次参考）

**领域分类**：按业务领域建立子目录（如按行业/业务线分），一个领域积累≥3个案例后，提炼共性为方法论笔记。

**驱动 Skill**：`project-retrospective-pm` 复盘后自动触发案例沉淀评估。

### 定期清理

每季度或按需执行一次，由 PM 手动触发：

1. **有效期检查**：政策法规是否有版本更新，标准是否过期
2. **转换时效检查**：二进制文件是否更新，转换产物是否需要重新生成
3. **案例库盘点**：是否有已完成项目未沉淀案例
4. **膨胀控制**：过期的 MD 标记 `~~已过期~~`，标注替代版本链接；确认无价值的直接删除
5. **交叉引用检查**：已过期内容的引用是否已更新指向

## 第四部分：协作规范

## 十一、文档生成纪律（10条）

撰写或编辑任何项目文档时，AI 必须遵守以下校验规则：

### 生成时

1. **完整性校验**：源文档有 N 项，产出不得少于 N 项；如有合并或省略，必须显式说明原因
2. **示例≠定义**：源文档中"如""举例""假设"等示例数值，汇报中必须还原为规则描述，不得将示例写死为定义
3. **最小信息密度**：每个功能点至少保留"做什么+怎么用/为什么"两层信息，不了解项目的人也能看懂
4. **角色必带边界**：每个角色首次出现时，用一句话交代"能做什么+不能做什么"
5. **流程闭环校验**：每个功能必须有"触发条件→操作步骤→执行结果"的完整闭环

### 编辑时

6. **编号一致性**：插入、删除、移动编号项后，必须全局刷新后续编号和交叉引用
7. **编辑后自查**：检查重复/残留段落、断链引用、编号不连续，发现即修

### 输出时

8. **中文编码处理**：解压含中文文件名的 zip 时，优先使用 cp437→gbk 解码方案

### 查询时

9. **先查知识库**：遇到 B 端 SaaS 的角色设计、权限设计、注册流程问题时，先查知识库再回答
10. **生成完成后按以上逐条自查**：发现遗漏立即修正

## 十二、智能体角色与边界

### 能做什么

| 职责 | 说明 | 依赖 |
|------|------|------|
| **文件管理** | 自动归类新文件到对应目录，创建缺失的 `_index.md`，更新索引 | Skill: `project-intake-pm` |
| **信息提炼** | 从会议记录中提取核心内容、对项目的影响、关键决策信号 | Skill: `project-intake-pm` |
| **文档生成** | 按模板撰写 PRD、设计规格、会议纪要、评审纪要 | Skill: `prd-writer`、`proto-spec-generator` |
| **原型输出** | 根据设计规格生成可交互的 HTML 原型 | Skill: `prototype-html-pin` |
| **架构可视化** | 生成架构图 HTML | Skill: `architecture-diagram` |
| **质量校验** | 执行"文档生成纪律"10 条校验，发现编号跳跃、断链引用、完整性缺失 | 内置规则 |
| **上下文加载** | 接到任务时自动读取关键决策、会议记录、案例库，确保输出基于最新共识 | 内置规则（Agent 定义） |
| **复盘驱动** | 项目阶段完成后系统性复盘，识别提示词缺陷、知识库缺失，驱动五级修复 | Skill: `project-retrospective-pm` |
| **知识库维护** | 将项目经验反向沉淀到大知识库：新增案例、更新方法论、补充术语、维护二进制索引 | Skill: `project-intake-pm`、`project-retrospective-pm` |

### 不能做什么

- **不做决策**：草拟关键决策默认标注"待确认"，不自动写"已确认"。技术选型、范围调整、商务判断等最终决策权在 PM
- **不替代评审**：产出 PRD/原型后，不自行判断"通过"，需 PM 或团队正式评审
- **不跳过上下文**：不凭记忆或猜测工作，必须先读关键决策和会议时间线再动笔
- **不覆盖源文件**：处理二进制文件时，原始文件始终保留在 `_source/`，仅生成转换产物

### 协作模式

```
PM 提供输入              智能体处理                PM 审阅确认
  ─────────           ─────────────           ─────────────
  会议内容      →      五步入库 + 提炼        →    确认关键决策
  需求方向      →      PRD 撰写               →    需求评审
  确认的需求    →      设计规格 + 原型         →    设计评审
  项目完成      →      复盘报告 + 修复建议     →    确认并执行修复
```

核心原则：**智能体加速执行，不替代判断。** 所有决策必须经过 PM 审阅确认才能生效。
