# 助理上手指南

> 本文是给新加入工作区的助理看的操作手册。读完能独立搭好环境、跑通第一个任务、知道自己能改什么不能改什么。
> 规则依据见同目录《项目工作流规范.md》和项目根 `AGENTS.md`，冲突时以那两份为准。

---

## 1. 你的角色边界（先读这一段）

本工作区采用**消费型协作**起步，意思是：

- 通用资产（`01-通用规则/`、`03知识库/`、`.ai/`、`AGENTS.md`）的**唯一权威源是项目负责人**。你只能读、只能 pull，不能直接修改、不能 push 这些目录的任何文件。
- 你的工作产出写在分配给你的**项目目录**里（通常是 `02当前项目/<某项目>/02-迭代/<某版本>/` 下的需求、规格、原型）。这部分你全权负责。
- 如果你发现通用规则有错、或沉淀了好经验想回流，**走提议流程**：把想改的内容写成一段文字发给负责人，由负责人判断是否入库。不要自己改。

这条边界是为了保证规则不失真。规则一旦多人随意改动，AI 加载的上下文就会互相打架，整套体系会很快崩。在你被验证为可以共建之前，守住只读。

---

## 2. 环境搭建（一次性，约 30 分钟）

前置条件：会基本命令行操作（cd、ls、git 三条命令），有一台能联网的电脑。不会 git 的，先花一小时学 `clone / pull / add / commit / push` 五个命令。

### 2.1 克隆仓库

```bash
git clone git@github.com:wsdlp46/PRDCraft.git 我的项目
cd 我的项目
```

如果提示没有权限，找负责人开通仓库的读权限。

### 2.2 安装 AI 工作流资产

在项目根目录执行：

```bash
bash .ai/install.sh
```

脚本会自动检测你电脑上装了哪个 AI 工具（ZCode、Claude Code、Codex），把 skills 和 agents 软链到对应的全局目录。看到"共链接 N 个 skill"即成功。

### 2.3 准备 AI 工具账号

这是**唯一需要花钱的环节**，也是能否顺畅工作的关键：

- **ZCode（推荐）** 或 **Claude Code**：skill 按关键词自动触发，体验完整。需要自己的 API key 或订阅账号。
- **Codex（OpenAI CLI）**：免费，但 skill 不会自动触发，每次都要手动 `@.ai/skills/xxx/SKILL.md` 引用，效率低，不建议作为主力。

账号问题找负责人确认怎么解决，不要默认用别人的 key。

### 2.4 验证安装成功

打开你的 AI 工具，把工作目录指到这个项目，然后说一句"开始新版本"。如果 AI 自动触发了 `project-iteration` 这个 skill（会开始跟你确认版本号、要复制哪一版的资产），说明环境通了。如果没有触发，先检查 install.sh 是否跑过、AI 工具是否重启过。

## 3. 试点任务（验证你能不能用这套体系）

环境搭好后，不要急着接真活。先用一个隔离的试点任务证明自己跑得通。找负责人要试点目录路径（通常是 `_sandbox/` 或某个练习项目），完成下面三件事，全程**独立操作，不手把手请教**，卡住超过 30 分钟再问。

### 任务一：跑通版本初始化

对 AI 说"开始新版本"，按 skill 引导创建一个 V0.1.0 测试版本目录骨架。验收标准：

- 生成了 `02-迭代/V0.1.0/` 完整目录结构（01-原始需求 到 06-变更记录 六个子目录）
- 生成了 `_index.md`，状态字段齐全
- 没有报错、没有半成品目录残留

**这个任务验证你会不会跟 AI 协作**。如果你不知道该对 AI 说什么、说了之后 AI 没按预期走也判断不出来，说明还需要练思维方式，光看文档没用。

### 任务二：改一个原型页面

照着试点目录里的一份规格文档，让 AI 修改对应的 HTML 原型页面（用 `prototype-html-pin` skill）。验收标准：

- AI 改完后用 `_scaffold/` 里的 build.sh 跑过构建，无报错
- 改动符合规格文档描述，没有多改、没有漏改
- 产出符合 `AGENTS.md` 的写作纪律（功能描述 67 到 120 字、无 emoji、无箭头符号）

**这个任务验证你能不能把规格翻译成 AI 听得懂的指令**。卡在"不知道怎么描述要改什么"是正常的，多练；卡在"看不出 AI 改得对不对"是危险的，说明对产品本身的理解还不够。

### 任务三：写一段功能描述

挑一个你熟悉的功能（比如登录、列表筛选），写 200 字左右的功能描述给 AI，让它按本工作区的写作风格输出。然后对照 `AGENTS.md` 的"文档生成纪律"十条自查。

**这个任务验证你的文字基本功和对纪律的敏感度**。如果自查后仍被负责人挑出 3 条以上问题，说明需要先熟读纪律再上手。

### 三个信号，对应三种结论

| 卡点 | 结论 |
|------|------|
| 卡在 git、命令行、安装脚本 | 能教，成本不高，过几天就会 |
| 卡在"不知道让 AI 干什么"、看不出 AI 输出对错 | 难教，这是思维方式问题，需要更长周期或换岗位匹配度 |
| 能跑通但产出质量差 | 可培养，靠 AGENTS.md 纪律和 review 慢慢收敛 |

试点不是考试不及格就淘汰，是让双方都看清差距在哪。

## 4. 日常工作节奏

环境通了、试点过了，进入正常协作。每天的节奏就四步：

1. **开工先 pull**：`git pull` 拉取负责人对通用资产的最新改动。这一步不能省，否则你用的规则是旧的，AI 会按旧规则产出。
2. **干活**：在自己的项目目录里写需求、画原型、跑 skill。产出只落在自己的目录。
3. **收工前 push 自己的产出**：`git add` 你改的项目目录、`commit`、`push`。**不要 add 通用资产目录**，哪怕你不小心改了也要 `git checkout` 还原。
4. **遇到通用资产要改**：写一段提议文字发给负责人，等审批。不要自己 commit。

push 频率：**每天至少一次**，不要攒一周才 push。小步快跑便于负责人 review，也避免冲突堆积。

---

## 5. 红线（碰了会出大问题）

1. **不经过审批修改 `01-通用规则/`、`03知识库/`、`.ai/`、`AGENTS.md`**。这是最重的红线，违反一次信任就崩。
2. **不 pull 直接干活**。用旧规则产出，等于白干。
3. **用别人的 API key**。账号要独立。
4. **把含密钥、客户数据、未公开商业信息的文件 push 到 gitee**。gitee 是半公开仓库，敏感信息一律放本地 `_附件/` 不入库。
5. **跳过 skill 直接让 AI 写**。skill 是这套体系的执行细节载体，绕过 skill 等于绕过规范，产出不可控。

---

## 6. 卡住了怎么办

按这个顺序自救，别一卡就问：

1. **先读文档**：`AGENTS.md`、本指南、`01-通用规则/项目工作流规范.md`、`.ai/TOOL-MAPPING.md`。八成的问题在这四份里有答案。
2. **再看 skill**：`.ai/skills/<skill名>/SKILL.md` 里写了这个 skill 干什么、怎么触发。卡在某个阶段就去读对应 skill。
3. **查知识库**：`03知识库/工作案例库/_index_AI匹配.md` 是紧凑索引，按关键词找案例路径。
4. **以上都看过还没解决**：把问题、你已经试过什么、报错截图整理清楚，再问负责人。问得越具体，答得越有用。

| 目录 | 能否改 | 说明 |

| `02当前项目/<你的项目>/` | 能 | 你的主战场，全权负责 |

| `03知识库/` | **不能** | 只读，改走提议 |
| `.ai/` | **不能** | skills/agents 源文件，只读 |
| `AGENTS.md` | **不能** | 工作区级指令，只读 |
| `00主页.md`、`docs-site/` | **不能** | 文档站，由负责人维护 |
