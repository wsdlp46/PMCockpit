# 交互式HTML产品原型生成（Pin 标注版）

根据产品需求，生成完整的、可直接运行的单文件HTML产品原型。

与原版 prototype-html 的核心差异：用**统一红色编号圆点**替代 SVG 贝塞尔连线，解决连线交叉、偏移、远距离不可读的问题。同时支持多状态组件切换演示和标签显隐控制。

## 核心工作流程

1. **理解需求** - 明确用户需要展示的产品功能和界面元素。若提供了规格文档（.md），通读全文。若项目中存在设计规范文档，读取获取配色、字号、组件尺寸等参数。
2. **风格判定** - 根据页面类型确定视觉方案：管理后台/B端 → 读取 `01通用/设计库/web后台-design.md` 获取 CSS 基线参数；门户首页/C端 → 读取 `01通用/设计库/web门户-design.md`；移动端H5 → 读取 `01通用/设计库/手机端-design.md` 或 `手机端-极简黑白-design.md`；数据大屏 → 深色底+卡片矩阵。同时读取 `01通用/设计库/设计约定.md` 获取共享约束。若用户未明确页面类型，默认按管理后台处理。
3. **规格匹配** - 从规格文档提取UI元素清单：过滤出表格列、卡片字段、按钮、弹窗层级、筛选条件。只做规格明确要求的元素，不擅自添加。不确定的枚举值只放第一个并标注待确认。
4. **设计标注** - 从界面中选定 3-5 个关键元素进行标注，规划每个标注在左侧界面上的目标元素位置，以及右侧说明区的对应说明条目。
5. **编写代码** - 生成符合所有技术规范的完整HTML文件。
6. **交付前自查** - 执行下方检查清单，逐项打勾通过后方可交付。

> 若为**已有原型迭代**（非新建），先执行下方「原型迭代工作流」，再执行交付前自查清单。

### 原型迭代工作流（修改已有原型时执行）

当用户要求修改已有原型页面时，必须同步执行以下流程，防止改了原型但遗漏规格和需求更新。

**第一步：归档旧版**

修改前先把当前原型文件拷贝到项目归档目录，文件名加日期后缀：
```
原型/HTML/O2d-提现审核-原型.html
  → 归档/原型旧版/O2d-提现审核-原型-20260619.html
```

**第二步：边改边记变更笔记**

在项目根目录 `_temp/变更笔记/` 下创建变更笔记文件（命名：`变更笔记_页面编号_YYYYMMDD.md`；若目录不存在则创建），每完成一处修改就追加一行。笔记格式为简短要点，自己能看懂即可：

```markdown
# O2d 提现审核 — 变更笔记（2026-06-19）

- 提现审核页拆成三阶段tab切换，不是页面跳转
- 待开票状态：代理端加上传发票按钮，运营端这阶段只读
- 发票退回要填原因，做了下拉+手填组合
- 列表状态Tag从3种变成6种，颜色重排
- 待打款阶段保留原有转账凭证上传，未改动
```

> 笔记只记录"改了什么"（界面/交互/业务规则变化），不写技术实现细节（CSS类名、JS逻辑、DOM结构等）。

**第三步：定稿后过防遗漏清单**

原型改完且用户明确确认效果后（如回复"可以了"/"定稿了"/"就这样"等），逐项检查以下7类容易遗漏的场景：

| 检查项 | 具体检查内容 |
|--------|-------------|
| 空状态 | 列表无数据时显示什么？搜索无结果时？ |
| 错误状态 | 上传失败、网络超时、OCR识别失败、表单校验不通过 |
| 权限边界 | 当前角色能看到什么、不能看到什么？是否有越权字段暴露？ |
| 状态回退 | 流程回退后界面恢复成什么状态？有无引导提示？ |
| 数量边界 | 列表上限、分页边界、金额为0、规则数到达上限 |
| 并发情况 | 多人同时操作、操作进行中状态变更、重复提交 |
| 消息通知 | 每个状态变更是否有对应的消息推送需求？若需求文档中未明确说明消息通知的触发时机和内容，**中断执行，将检查结果和待确认项汇报给用户，待用户确认后再继续。** |

发现遗漏直接补到原型和变更笔记中。

**第四步：列回写清单，用户确认后再执行**

原型定稿、用户确认效果后，不要直接回写。先整理一份回写清单交给用户确认：

```markdown
## 回写清单 — O2d 提现审核（2026-06-19）

### 需要更新的规格文档
1. `规格文档/O2d-提现审核-原型设计规格.md`
   - 新增：发票验证阶段页面结构（阶段二）
   - 修改：状态定义从3种扩展为6种
   - 修改：操作菜单适配新状态
2. `规格文档/O2c-分润与提现-原型设计规格.md`
   - 修改：提现列表状态列新增待开票/待验票/待打票
3. ...

### 需要更新的需求文档
1. `需求文档/代理结算与提现——合规细化需求V1.1.md`
   - 确认无遗漏（已是最新）

### 需要更新的台账
1. `需求文档/需求变更台账.md` — 批次01状态改为"规格已更新"
```

用户确认清单后，再逐个执行回写。

**第五步：执行回写**

按确认的清单逐个更新规格文档和需求文档。每更新完一个文件，在清单中标记完成。

**第六步：清理变更笔记**

规格和需求都更新完成后，变更笔记已完成使命。将笔记内容合并到规格文档的变更记录区，然后删除临时笔记文件。合并格式示例：

```markdown
## 变更记录

| 日期 | 页面编号 | 变更内容 |
|------|----------|----------|
| 2026-06-19 | O2d | 状态定义从3种扩展为6种，新增待开票/待验票/待打票 |
| 2026-06-19 | O2d | 发票退回增加退回原因填写（下拉+手填组合） |
```

若规格文档已有变更记录区，追加新行即可；若没有，在文档末尾新增"## 变更记录"区域。

> **关键原则**：变更笔记是"决策现场记录"，必须在改原型的同时写，不能事后凭记忆补记。遗漏不是能力问题，是记录时机问题。回写前必须列清单经用户确认，不直接动手。

### 交付前自查清单

逐项对照规格文档检查（新建和迭代均需执行）：

**字段一致性：**
- [ ] 表格/卡片的列名、列顺序、字段是否与规格一一对应，无多余、无遗漏？
- [ ] 所有输入框label、下拉选项、状态标签文本是否与规格一致？
- [ ] 只读/可编辑/条件显示等字段状态是否与规格一致？

**交互完整性：**
- [ ] 每个操作项是否覆盖了规格列出的所有按钮？是否有规格未列出的多余按钮？
- [ ] 弹窗层级（L1/L2/L3）是否与规格标注一致？
- [ ] 筛选栏的筛选项和控件类型是否与规格一致？

**业务规则：**
- [ ] 规格中明确的字段间约束关系是否实现？（如A是B的子项还是独立字段）
- [ ] 规格中明确的业务参数是否准确？（参见项目知识库中的术语定义）

**说明文档同步：**
- [ ] 页面迭代后，右侧功能说明是否与当前界面一致？标注编号是否仍然一一对应？
- [ ] 对于有多个状态的组件，状态切换按钮覆盖了所有需要演示的状态？

> 注：具体的字段名、列顺序、业务参数以用户提供的规格文档和项目知识库为准，本清单仅检查"是否做到了"，不预判"应该是什么"。

### 多页面系列规则

当原型涉及多个同系列页面：
- 同一端侧的侧边栏菜单项应完全一致，当前页标记`active`类。
- 菜单项应为可点击的`<a href="文件名.html">`链接。
- 新增页面时同步更新同系列所有已有页面的侧边栏。

## B 端原型设计准则（交互层）

画 B 端管理后台原型时，以下 6 条准则是功能与交互层面的硬约束，优先于具体布局规范执行：

1. **分类前置导入**：批量导入弹窗必须包含上级分类选择项，不允许先导入再归类，避免数据孤立
2. **操作按钮聚合**：同类操作（增删改查、导入导出、批量处理）物理聚合到同一区域，减少视线和鼠标来回切换
3. **交互流程极简**：能在一屏内完成的操作不分步弹窗，降低交互深度，避免"点开三层才能完成任务"
4. **显性化查看入口**：操作列首位固定放"查看"按钮，满足读写分离习惯，查看频率远高于编辑
5. **业务状态去冗余**：无跨角色审批流时砍掉中间态（如"待提交""已暂存"），只保留有业务意义的终态
6. **弹窗层级规范**：严格区分 L0 页面、L1 侧滑窗、L2 居中弹窗、L3 确认弹窗，蒙版不穿透（上层蒙版必须遮住下层，不允许同时看到两层弹窗）

## 布局规范（严格执行）

### 整体结构

- **左右分栏布局**
  - 左侧：占页面宽度的78%，展示产品主体界面内容
  - 右侧：占页面宽度的22%，展示对应功能的详细说明文档
- **分隔线**：左右两栏之间有清晰的垂直分隔线
- **滚动同步**：左右两栏各自独立滚动，互不影响

### 色标编号标注方案（替代传统连线）

不再使用 SVG 贝塞尔曲线连线。采用「色标圆点 + 数字编号」方案建立左右对应关系。

**标注规则：**
- 从左侧界面中选定 3-5 个关键元素进行标注
- 每个被标注元素添加 `data-anchor="N"` 属性（N 为数字 1-6），以及 `class="anchor-target"`
- JS 动态创建彩色数字圆点（`position: fixed; z-index: 100`，挂载到 `document.body`），贴在目标元素右边缘外侧、垂直居中
- 圆点颜色统一使用红色 `#F53F3F`，不区分编号颜色

**右侧说明区：**
- 每条说明条目前方放置同色同号的圆点，与左侧一一对应
- 说明条目结构：`<span class="doc-pin" style="background:色值">编号</span>` + 标题 + 描述文字

**多状态组件演示：**
- 对于存在多种视觉状态的 UI 组件（如认证状态卡片、审核状态标签等），在右侧对应说明条目中嵌入状态切换按钮
- 按钮结构：`<div class="state-switcher"><button data-state="状态值" class="active">状态名</button>...</div>`
- 点击按钮时同步切换左侧组件的 CSS 类名和右侧描述文字，方便演示不同状态下的视觉效果

**标签显隐控制：**
- 右侧面板顶部放置一个开关按钮，一键显示/隐藏所有左侧标注圆点
- 开关默认开启，关闭后圆点全部隐藏（`display: none`），方便截图展示纯界面

**交互效果：**
- 鼠标悬停左侧圆点或标注元素 → 右侧对应说明行背景高亮 + 左边框变色
- 鼠标悬停右侧说明条目 → 左侧对应标注元素加虚线边框高亮
- 双向联动，无需物理连线

**技术要点：**
- 圆点用 `position: fixed` 挂载到 `document.body`，确保层级高于左右分栏中的所有内容
- 在目标滚动容器和 `window` 上监听 `scroll` 和 `resize` 事件，实时重算圆点位置
- 圆点通过 `target.getBoundingClientRect()` 获取目标元素的视口坐标，`rect.right + 2` 作为 left，`rect.top + rect.height / 2` 作为 top

## 响应式设计规范

### PC端
- 保持上述左右分栏布局

### 手机端
- 自动切换为上下布局
- 上方展示产品主体界面，下方展示功能说明
- 标注圆点自动隐藏
- 在每个元素下方直接显示简短说明
- 提供"切换视图"按钮，允许用户在"仅界面"和"界面+说明"之间切换

## 技术栈要求

1. **纯前端实现**：使用纯HTML、CSS和原生JavaScript，不依赖任何外部框架
2. **样式**：使用Tailwind CSS v3，通过CDN引入
3. **图标**：使用Font Awesome图标库，通过CDN引入（可选，根据界面需要决定是否引入）
4. **代码结构**：
   - HTML结构语义化，使用适当的标签
   - CSS样式内联在`<style>`标签中
   - JavaScript代码内联在`<script>`标签中
   - 所有代码整合在一个HTML文件中，无需外部资源

## 样式与视觉规范

优先读取设计文档，按优先级逐层覆盖：
1. 读取 `01通用/设计库/` 对应端态的 design.md，获取默认主色、字号、间距、表格行高、按钮样式等 CSS 基线参数
2. 若项目有自定义主色，覆盖基线默认值
3. 读取 `01通用/设计库/设计约定.md`，获取跨项目共享约束（圆角规范、阴影层级、画板宽度）
4. 读取 `01通用/设计规范/Web应用设计规范.md`（或政务APP设计规范.md），确认品牌级组件参数

以下为原型特有的补充规则（设计文档不涵盖的部分）：

### 原型特有布局
- **左右分栏**：左侧产品界面 78%，右侧功能说明文档 22%
- **右侧说明区**：琥珀色系（`#FFFBEB` 底色 + `#fde68a` 边框），与左侧内容区形成视觉区分
- **标注圆点**：22px 圆形，`position: fixed; z-index: 100`，白色数字，投影 `0 2px 6px rgba(0,0,0,0.25)`

### 设计规范未覆盖时的默认值
仅当项目无设计文档时使用以下默认值：
- 品牌色 `#165DFF`，成功 `#00B42A`，错误 `#F53F3F`
- 字体：系统默认无衬线字体
- 间距：以 8px 为基本单位
- 整体风格：简洁、现代、专业

## 输出要求

1. 输出完整的HTML文件代码
2. 代码开头添加简短注释，介绍原型的功能和使用方法
3. 确保所有标注圆点位置准确，滚动和 resize 后不偏移
4. 确保标注编号与右侧说明一一对应，颜色一致
5. 确保在不同屏幕尺寸下都能正常显示和交互

## 注意事项

- 标注圆点必须在 DOM 渲染完成后创建，使用 `DOMContentLoaded` 或在 `</body>` 前执行初始化脚本
- 圆点用 `position: fixed` 挂在 `document.body`，避免被父级 `overflow: hidden` 裁剪
- 在包含滚动容器（如表格区域）的页面中，需对滚动容器和 `window` 同时监听 `scroll` 事件重算位置
- `window` 的 `resize` 事件也需要重算圆点位置
- 所有 `data-anchor` 属性和 ID 命名要清晰，便于脚本定位
- 状态切换按钮改变组件样式后，需延迟 50ms 调用位置重算（DOM 可能因样式变化导致元素尺寸微调）
