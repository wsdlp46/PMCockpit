# TAPD 需求上传

## 前置条件

执行任何步骤前，先向用户确认：
1. **范围**：用户说「企业端页面」→ 去读所有 `E`/`O` 前缀规格文档；说「代理端」→ 读 `A` 前缀；说「公共」→ 读 `P` 前缀
2. **腾讯文档链接**：当前版本 MVP 需求文档的腾讯文档链接
3. **端侧**：PC Web / 微信小程序，需要用户指定

## 工作流（5步）

### Step 1：加载项目上下文

按以下顺序读取：
1. `_index.md` → 了解项目整体结构
2. `01-项目输入/会议纪要_时间线.md` → 检查是否有会后更新影响页面
3. `01-项目输入/关键决策.md` → 检查是否有决策影响页面范围

### Step 2：读取需求文档和规格文档

按用户给定的范围，读取两类文档：

**需求文档**（必读）：
- 文件名通常为 `需求文档/<项目名>——MVP需求文档V*.md`
- 从中获取：功能详述（F1/F2/F3章节）、页面边界（PC/小程序分工表）、消息通知规则

**规格文档**（逐页读）：
- 位置：`规格文档/` 下，按编号前缀匹配
- 读取每一页后检查完整性：至少需要「功能权限」「交互规则」「数据来源」三个字段存在且有实质内容
- 如果某页缺失 → **立即停止，向用户报告**：「页面XX的规格文档缺少YY字段，无法生成TAPD需求。请先补齐后重试。」

**页面分类规则**：

| 用户说法 | 读这些规格文档 | 端侧 |
|------|------|:---:|
| 企业端PC页面 | E* 前缀（E0/E1/E3/E4/E6/E7/E9/E10/E11/E12） | PC Web |
| 管理端页面 | O* 前缀（O1/O1a/O1b/O2/O2a/O2c/O2d/O3/O6） | PC Web |
| 企业端小程序 | MP-E* 前缀 | 微信小程序 |
| 代理端小程序 | MP-A* 前缀 | 微信小程序 |
| 公共页面 | MP-P* 前缀 | 微信小程序 |

### Step 3：生成规划文档

从规格文档的对应字段提取内容，填入 `assets/plan-template.md` 模板：

| 模板字段 | 从规格文档提取 | 提取规则 |
|------|------|------|
| 编号 | 页面编号（如 E0、MP-A1） | 直接取 |
| 页面名称 | 页面名称 | 直接取 |
| 端侧 | PC Web / 微信小程序 | 由 Step 2 对照表判断 |
| 数据权限 | 功能权限章节 | 仅 PC 端填写：本企业 / 本部门 / 全平台 / 代理名下 |
| 功能权限 | 功能权限章节 | 仅 PC 端填写：查看 / 查看+导出 / CRUD / 审核操作 |
| 面向用户 | 角色分析章节 | 直接取 |
| 定位 | 功能概述第一句 | 提取一句话 |
| 前置与跳转 | 交互规则章节 | 进入方式 + 特殊跳转（侧边栏导航不写） |
| 页面元素 | 交互规则 + 功能权限 | 按区域分组，每行：区域/元素名/交互行为/数据来源 |
| 状态分支 | 交互规则中的条件判断 | 条件→展示/行为，无则跳过 |
| 相关文档 | 用户提供的腾讯文档链接 + 来源章节号 | 如「MVP需求文档 第三章 F1.6」 |

**数据权限枚举**（仅PC端页面使用）：

| 值 | 含义 |
|------|------|
| 本企业 | 只能看自己企业的数据 |
| 本部门及子部门 | 部门管理者看下属，当前项目暂不用 |
| 全平台 | 运营看所有企业/代理的数据 |
| 代理名下 | 代理看自己客户的数据 |

**功能权限枚举**（仅PC端页面使用）：

| 值 | 含义 |
|------|------|
| 查看 | 纯只读，无操作按钮 |
| 查看+导出 | 只读 + Excel导出 |
| 新增+编辑+删除 | 完整CRUD |
| 审核操作 | 通过/驳回审批 |
| 配置操作 | 设置类页面，如企业设置、分润比例 |

**页面元素描述规则**：
- 按「区域」分组（顶部/筛选栏/列表/行内操作/弹窗/底部等）
- 每行一句话讲清楚「什么元素 + 用户怎么操作 + 系统怎么响应」
- 数据来源标注接口名或微信开放能力

**规划文档保存路径**：`01-项目输入/TAPD上传规划-{{端侧}}需求.md`

### Step 4：拉取 TAPD 配置并上传

按以下顺序操作：

**4.1 拉取配置**：
```
get_stories_fields_info(workspace_id)
→ 提取：category_id 选项（用户可能已建，也可能需要新建）
→ 提取：iteration_id（当前迭代）
→ 确认：priority_label（默认 High = P0）
```

**4.2 映射分类**：
将规划文档中的端侧映射到 TAPD category_id。如果用户说「分类已建好」，按以下默认映射：

| 规划端侧 | TAPD 分类 | 典型 category_id |
|------|------|------|
| PC 企业端 | 企业端 | 需从 fields_info 查 |
| PC 管理端 | 管理端 | 需从 fields_info 查 |
| 小程序公共 | 主页 | 需从 fields_info 查 |
| 小程序代理端 | 代理小程序 | 需从 fields_info 查 |
| 小程序企业端 | 企业小程序 | 需从 fields_info 查 |

**如果分类映射不匹配 → 暂停，告知用户需要手动创建对应分类。**

**4.3 逐条上传**：

每页一条 `create_story_or_task`：

```
{
  "workspace_id": <项目ID>,
  "name": "[端侧] 页面编号 页面名称",
  "options": {
    "entity_type": "stories",
    "priority_label": "High",
    "iteration_id": "<当前迭代ID>",
    "category_id": "<映射后的分类ID>",
    "description": "<h3>1. 页面信息</h3>..."
  }
}
```

**关键约束**：
- 不建父需求和容器，所有需求平级独立
- `description` 必须放在 `options` 内，不能放在根层
- 描述内容使用 HTML 标签：`<h3>` `<table>` `<tr>` `<td>` `<th>` `<p>`
- 逐条创建，完成后告知用户共创建了多少条

**4.4 修复错误**：
如果出现分类错误或父子关系错误：
- `update_story_or_task` 设置 `is_archived:"1"` 隐藏旧需求
- 重新创建正确的新需求
- TAPD 没有 delete API，归档是最接近删除的操作

### Step 5：验证

上传后用 `get_stories_or_tasks` 按分类查询，确认：
- 条数正确
- 分类正确
- 迭代正确

## 参考案例

### 关于父需求/容器用法

父需求承载「做什么、为什么做、验收标准」，子需求承载「怎么做」。

**何时用**：一个 Feature 跨多个页面、有统一验收目标。例如：

```
[Epic] 企业账号体系（父需求）
  描述：目标、范围、验收标准、关联文档
  ├── [Story] E0 企业概览
  ├── [Story] E3 组织架构管理
  └── [Story] E4 员工管理
```

**何时不用**：页面之间没有强逻辑关联、只是同一分类下的独立页面。默认不建父子关系，除非用户明确要求。

## 常见陷阱

1. **description 放错层级** → `create_story_or_task` 的 description 必须在 `options` 对象内，放根层会报 `must NOT have additional properties`
2. **分类ID不可编造** → 必须从 `get_stories_fields_info` 返回值中取，每个项目的 ID 不同
3. **规格文档不完整** → 必须逐页检查，缺字段立即停止并预警，不跳过继续
4. **旧需求删除不了只归档** → 用 `is_archived:"1"`，告知用户去 TAPD 回收站手动清除
5. **全栈研发不需要前后端衔接细节** → 页面元素描述聚焦「用户看到什么、操作了什么」，不写前后端交互细节
