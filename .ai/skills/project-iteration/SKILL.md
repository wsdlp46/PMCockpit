# project-iteration — 迭代版本管家

## 目的

管理迭代型项目的**版本生命周期**：从"开始新版本"到"版本收尾"。让新版本初始化标准化（目录骨架、资产复制、原型剥离一键完成），让版本收尾不留尾巴（状态校正、索引更新、总览同步）。

## 触发条件

- 用户明确说"开始 V1.0.X 迭代""新建版本""启动下一迭代"→ 执行**版本初始化**
- 用户说"版本收尾""结项""归档""收个尾"→ 执行**版本收尾**
- 用户说"_index 不对/索引同步/状态更新"→ 执行**收尾中的 sync-index 步骤**

> 不主动建议"要不要新建版本"——由用户声明触发，避免过度工程。

---

## 前置依赖

- 项目遵循四层结构（通用/迭代/版本/目录），见 `01通用/项目工作流规范.md`
- 版本目录命名：`0X-V{MAJOR}.{MINOR}.{PATCH}`（如 `03-V1.0.2`）
- 版本内目录：`01-原始需求` ~ `06-变更记录`（无评审纪要，见规范）

---

## 流程一：版本初始化

用户声明开始新版本时执行。

### 第1步：确定版本号

- 查 `02-迭代/` 下已有版本，确定下一个版本号
- 主版本（功能迭代）增 MAJOR，小版本（优化/补丁）增 MINOR/PATCH
- 向用户确认版本号和本版本范围

### 第2步：创建版本目录骨架

```
02-迭代/0X-V1.0.X/
├── _index.md              ← 从上一版复制模板，清空待办，更新版本号
├── 01-原始需求/
├── 02-需求文档/
├── 03-规格文档/
├── 04-原型/
├── 05-TAPD上传规划/
└── 06-变更记录/
```

### 第3步：复制上一版资产

从上一版本**完整复制**（非仅差异）：
- `03-规格文档/` 全部规格 → 新版本 `03-规格文档/`
- `04-原型/` 全部 HTML + `_scaffold/` → 新版本 `04-原型/`
- `_index.md` → 改版本号、清空需求清单、状态重置为待启动

> **不复制** `01-原始需求` `06-变更记录`（每版本独立）。`02-需求文档` `05-TAPD` 视情况。

### 第4步：剥离原型旧标注

对新版本 `04-原型/` 下每个 HTML 调用 prototype-html-pin 的剥离脚本：

```bash
# 逐个剥离，得到干净骨架
cd 02-迭代/0X-V1.0.X/04-原型
for f in *.html; do
  bash <prototype-html-pin路径>/assets/strip-annotations.sh "$f" -o "$f" -y
done
```

剥离后页面保留界面、无标注无右侧说明。**剥离前的旧版带标注原型保留在上一版本目录**，不额外归档。

> 剥离脚本的具体规则由 prototype-html-pin skill 定义，本 skill 只负责牵引调用时机。

### 第5步：确认初始化完成

- 跑 `assets/sync-index.sh` 校正新版本 `_index.md` 状态（此时应全为待启动，资产已就位）
- 向用户报告：版本号、目录结构、复制了哪些资产、剥离了几个页面
- 提示用户可以开始需求撰写/规格/原型工作了

---

## 流程二：版本收尾

用户声明版本收尾/结项时执行。

### 第1步：校正状态索引

```bash
# 校正当前版本（和所有版本）的 _index.md 状态列
bash assets/sync-index.sh 02-迭代
```

确认状态与实际文件一致（规格完成/原型完成/待启动）。

### 第2步：更新项目总览

更新 `00-项目总览.md` 的迭代状态表：
- 本版本状态改为"已完成"或"待评审"
- 如有下一版本计划，补充到状态表

### 第3步：更新迭代索引

更新 `02-迭代/_index.md` 的版本清单（版本号/状态/说明）。

### 第4步：确认收尾完成

- 向用户报告：状态校正结果、总览/索引更新情况
- **主动提醒复盘**：建议用户执行复盘（`project-retrospective-pm` skill），覆盖本版本的流程问题、工具链规范度、可复用沉淀

> 归档逻辑：版本目录只留最终交付物。但本流程**不主动移动过程文件**——上一版本的带标注原型保留在原版本目录即可，过程草稿若用户要求清理才移入 `归档/`。

## 资产：工具脚本

### sync-index.sh（`assets/sync-index.sh`）

扫描版本目录，按文件存在情况自动校正 `_index.md` 的目录导航状态列。

**判定逻辑**：
- `02-需求文档/` 有 .md → "需求完成(N)"
- `03-规格文档/` 有 .md → "规格完成(N)"
- `04-原型/` 有 .html → "原型完成(N)"
- `05-TAPD上传规划/` 有 .md → "TAPD完成(N)"
- 全部完成 → 版本状态升级"待收尾"

**用法**：
```bash
./sync-index.sh 02-迭代/01-V1.0.0     # 校正单个版本
./sync-index.sh 02-迭代               # 校正该迭代下所有版本
```

**何时用**：
- 版本初始化完成后（确认骨架就位）
- 版本阶段性完成后（如原型画完、规格写完）
- 版本收尾时（最终校正）

> 解决 _index.md 状态失真问题（导航标"待启动"但文件实际已存在，导致 AI 第一眼就是错的）。

## 边界（不做什么）

- ❌ 不画原型 → `prototype-html-pin`
- ❌ 不写规格文档 → `proto-spec-generator`
- ❌ 不做复盘 → `project-retrospective-pm`
- ❌ 不传 TAPD → `tapd-requirement-upload`
- ❌ 不做架构图 → `architecture-diagram`
- ❌ 不主动建议建新版本（用户声明才触发）
